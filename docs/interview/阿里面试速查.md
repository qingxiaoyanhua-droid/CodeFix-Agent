## 二、三层记忆系统（核心创新！面试加分项）

### 2.1 完整架构图

```
用户输入（有bug代码）
     │
     ▼
┌─────────────────────────────────────┐
│  L1 工作记忆（当前会话）              │
│  Bug类型 + 错误信息 + 测试结果        │
└─────────────────┬───────────────────┘
                  │
                  ▼
┌─────────────────────────────────────┐
│  L2 Reflection（守）— 失败教训库      │
│  本次/历史失败case分析成结构化JSON     │
│  bug_pattern + root_cause            │
│  failed_strategies + lesson_learned  │
│  检索：Embedding语义 + 关键词 + 信任加权 │
│  验证环：usefulness_score动态更新      │
└─────────────────┬───────────────────┘
                  │
                  ▼
┌─────────────────────────────────────┐
│  L3 Hermes Skill（攻）— 成功SOP库    │
│  成功修复抽取为可复用SOP              │
│  steps + trigger_keywords            │
│  触发门：复杂度≥1 或 大模型介入       │
│  增量打补丁（不重写整个skill）        │
│  过期降权：deprecation_score>0.7→stale│
│  验证环：success_count/failure_count │
└─────────────────┬───────────────────┘
                  │
                  ▼
        注入相关记忆 → 小模型CoT推理
```

### 2.2 L2 Reflection（守）— 失败教训库

**什么时候写入**：
- 3轮修复全部失败的case

**存储格式**：
```json
{
    "bug_pattern": "initialization_error",
    "root_cause": "total初始化为1而非0，导致累加结果多1",
    "failed_strategies": [
        "只改了return值但没改total初始值",
        "只加了判断但逻辑反了"
    ],
    "lesson_learned": "修复累加类bug必须先检查所有初始化语句",
    "usefulness_score": 0.85,
    "hit_count": 3,
    "last_updated": "2026-04-10"
}
```

**检索逻辑（三个维度同时检索）**：
1. Embedding语义相似度（Top-3）
2. bug_type关键词精确匹配
3. usefulness_score信任加权

**验证环**：
- 命中且成功 → usefulness_score += 0.1
- 命中但仍失败 → usefulness_score -= 0.2
- 低于0.2 → 软删除，不再优先召回

**上下文控制**：
- L2最多注入3条，总tokens ≤ 500
- 必须有≥1个bug_keywords重叠
- relevance_score < 0.65过滤

### 2.3 L3 Hermes Skill（攻）— 成功SOP库

**什么时候抽取**：
- 修复成功 且（复杂度≥1 或 大模型介入过）
- 简单bug（小模型一次过）不抽取，避免skill过载

**存储格式**：
```json
{
    "skill_pattern": "累加函数初始化修复",
    "trigger_keywords": ["sum", "total", "accumulate", "初始化"],
    "complexity_score": 2,
    "steps": [
        "Step1: 检查所有累加变量的初始化值",
        "Step2: 确认初始值应为0而非1",
        "Step3: 验证循环边界不越界",
        "Step4: 用边界测试用例验证（[],[1],[1,2]）"
    ],
    "success_count": 7,
    "failure_count": 1,
    "deprecation_score": 0.12,
    "last_updated": "2026-04-12"
}
```

**增量打补丁**：不重写整个skill，只追加/修正steps；failure_count增加时追加新的failed_attempt到steps末尾。

**过期降权**：deprecation_score > 0.7 → 标记stale；连续3次失败 → 硬删除。

### 2.4 面试话术

> "我设计了三层记忆系统，类比人的学习过程。L1是当前工作记忆；L2是反思笔记——把失败教训写成结构化JSON存入磁盘，下次遇到类似bug自动检索，usefulness_score动态更新；L3是技能卡片——把成功修复抽成SOP模板，只对复杂case抽取，避免skill过载。这两层不占显存，只在推理时选择性注入。"

**追问：L2/L3会不会上下文爆炸？**
> "三个硬约束：①token预算上限L2+L3不超过上下文30%；②必须有关键词重叠（≥1个bug_keywords）；③按relevance_score排序后截断。"

**追问：L2/L3指标怎么量化？**
> "核心指标是avg_attempts（平均尝试次数），反映L2避免重蹈覆辙的能力。hit_rate反映检索召回质量。"

## 三、Compile-First ReAct 架构（核心必背）

```
有bug代码 → 小模型(1.5B)生成CoT+修复代码
                ↓
        compile + 测试用例（Ground Truth）
                ↓
        PASS(60%) → 直接输出
        FAIL(40%) → 报错+CoT发给大模型(7B)
                        ↓
                大模型分析：哪步推理出错，给hint
                        ↓
                小模型基于hint重试（最多3轮）
                        ↓
                3次都失败 → 存入L2 Reflection
```

**为什么最多3轮？**
> "统计显示90%的可修复bug在前3轮内完成，第4轮及以后的修复成功率接近0。继续重试浪费算力。超过3轮的case进入L2反思记忆，为下次同类bug做准备。"

**关键数字**：
- 60% case首轮通过，大模型调用减少60%
- 平均延迟降低44%
- 最多3轮重试，不通过记录为bad case

## 五、GRPO Loss 公式（必背）

```
Step 1：对每个prompt，采样 G=4 个response
Step 2：计算reward  r_i = reward(completion_i)
Step 3：组内标准化 advantage
        mean = Σr_i / G
        A_i = (r_i - mean) / std

Step 4：clip保护策略更新幅度
        ratio = π_θ(token) / π_ref(token)
        clipped = clip(ratio, 1-ε, 1+ε)   ε=1.2
        L_policy = -min(ratio × A, clipped × A)

Step 5：KL约束防止策略崩溃
        L_total = L_policy - β × KL(π_θ || π_ref)   β=0.1
```

**追问1：为什么选GRPO而不是PPO？**
> "PPO需要额外Critic模型估算V值，显存要40GB，A100 32GB跑不起来。GRPO同组内互相做baseline，无需Critic，显存友好。"

**追问2：KL系数怎么选？**
> "一般0.05~0.25，我设0.1。太小→策略偏移约束不住；太大→训练太保守。也可以做KL warmup，前几个epoch用较大β逐步衰减到0.1。"

## 六、代码项目·六维Reward（权重相加=1.00）

> 注意：这是代码修复任务的六维！衡量CoT推理链完整性！不是会议总结的六维！不能混用！

| 维度 | 权重 | 信号来源 | 怎么算 | 衡量CoT哪一步 |
|------|------|---------|--------|------------|
| **格式** | 0.05 | CoT结构完整性 | `'<think>'`开始0.25 + `'</think>'`结束0.25 + 每步`[Step i:]`各0.125 + `<fixed_code>`各0.25 + 4步全齐额外+0.2 | 整体结构 |
| **语法** | 0.10 | compile()能过吗 | Python `compile()` 通过=1.0，SyntaxError=0.0 | 整体结构 |
| **AST结构** | 0.15 | 和参考答案结构相似 | Jaccard(节点类型集合)×0.6 + 函数签名匹配×0.4 | 整体结构 |
| **执行** | 0.30 | 测试用例通过率（最重要） | 通过数/总数，ACES风格加权 | 整体结构 |
| **步骤级PRM** | 0.25 | 7B对CoT每步打分 | FIPO加权[0.15, 0.20, 0.30, 0.35] | **每一步单独衡量** |
| **语义** | 0.15 | 和参考答案语义相似 | BGE embedding cosine similarity | 整体结构 |

> **六维核心逻辑**：一个正确的修复 = CoT四步全对 → 代码才能修对。六维分别对应推理链上的不同环节。步骤级PRM是唯一能追溯到CoT每步质量的维度，信息密度是困惑度的4倍。

**格式奖励详细打分规则**：
```python
score = 0.0
if '<think>' in text: score += 0.25
if '
</think>' in text: score += 0.25
for i in range(1, 5):
    if f'[Step {i}:' in text: score += 0.125  # 每个步骤标签
if '<fixed_code>' in text and '</fixed_code>' in text:
    score += 0.25
# 完整4步CoT额外奖励
if has_all_4_steps: score += 0.2
```

**ACES测试区分能力加权**（不同bug类型有不同的区分能力）：
```python
TEST_DISCRIMINATIVE_WEIGHTS = {
    "logic_error": 1.2,      # 高区分：正确vs错误差异明显
    "off_by_one": 1.2,       # 高区分
    "initialization": 1.1,    # 高区分
    "boundary": 1.1,          # 高区分
    "null_pointer": 1.1,     # 高区分
    "wrong_operator": 1.0,    # 中等
    "missing_check": 1.0,     # 中等
    "type_error": 0.9,        # 中等
    "syntax_fix": 0.6,       # 低区分：多数正确代码都pass
    "style": 0.5,
    "optimization": 0.5,
}
```

**FIPO步骤位置加权**（Step越靠后权重越高）：
```python
STEP_WEIGHTS = [0.15, 0.20, 0.30, 0.35]  # Step4权重最高
# Step1-2是推理过程，不直接决定答案
# Step3（Fix Strategy）决定最终代码长什么样
# Step4（Edge Case）是最后一道防线
```

**为什么不用困惑度做reward？**
> "困惑度衡量流畅度，不衡量正确性。流畅但错的代码perplexity很低，会被错误奖励。六维里步骤级PRM提供4个step-level信号，信息密度是困惑度的4倍。"

**追问：权重怎么定的？全程不变吗？**
> "执行奖励权重最高（30%），因为测试用例通过是客观ground truth。格式放在第一位，因为CoT结构断裂会导致步骤级奖励失效。目前全程不变——这是局限，未来应该按训练阶段自适应调整，或用AutoRL自动搜。"

## 八、RAG混合检索（必背）

```
Query → Bug Type Index（倒排，596条→85条同类型）
              ↓
        BM25关键词(60%) + FAISS向量(40%)
              ↓
        RRF融合（量纲无关，工业标准）
              ↓
        Cross-Encoder精排 → Top-3
              ↓
        同类型Bug Boost ×1.5
```

**RRF公式**：
> RRF_score(d) = Σ 1/(k + rank_i(d))，k=60，对各检索方法排名取倒数求和

**检索参数（完整）**：
```
BM25: k1=1.5, b=0.75（代码感知分词，snake_case/camelCase拆子词，保留运算符）
FAISS: IndexFlatIP + L2归一化 = 余弦相似度，维度768
Embedding: microsoft/unixcoder-base（768维），降级fallback: all-MiniLM-L6-v2（384维）
Cross-Encoder: cross-encoder/ms-marco-MiniLM-L-6-v2
Type Boost: 同bug类型×1.5
```

**追问：为什么BM25要60%权重？**
> "代码bug有精确关键词模式——'off_by_one'、'wrong_operator'，纯语义检索对精确匹配不敏感。BM25捕捉关键词，FAISS捕捉语义相似，RRF融合取两者之长。"

**GRPO完整loss代码**：
```python
def grpo_loss(log_probs, rewards):
    # Step 1-2: 组内标准化
    mean_reward = rewards.mean()           # 所有样本reward均值
    std_reward = rewards.std() + 1e-8    # 防止除零
    normalized_rewards = (rewards - mean_reward) / std_reward

    # Step 3: policy gradient with normalized rewards
    loss = -torch.mean(log_probs * normalized_rewards)

    # Step 4: KL散度约束（实际实现时加）
    # L_total = loss - β * KL(π_θ || π_ref)  β=0.1
    return loss
```

## 十二、公式速记

```
BM25:
score = Σ IDF(w) × tf×(k+1) / (tf + k×(1-b + b×dl/avgdl))
  k=1.5, b=0.75
  IDF = log((N-n+0.5)/(n+0.5))

RRF:
RRF_score(d) = Σ 1/(k + rank_i(d))  k=60

HNSW:
  搜索复杂度: O(log N)
  EF_construction: 越大→召回越高→构建越慢
  EF_search: 控制精度-速度权衡

KL散度（GRPO中）:
  reward_final = reward_raw - β × KL(π_θ || π_ref)   β=0.1
```

## 十三、面试高频问题回答

### Q1：为什么CoT用四步而不是更多？
> "实验对比了3/4/5/6步：3步时很多bug定位和根因混在一起，Step2经常溢出；5步和6步没有显著提升，但小模型上下文窗口和推理能力不足以支撑更多步，导致推理链断裂。所以4步是精度和召回的平衡点。"

### Q2：ReAct和Plan的区别？
> "Plan先生成完整计划再执行；ReAct是思考→行动→观察反馈的循环，边做边调整。代码修复用ReAct更合适，因为编译器和测试用例提供了实时反馈——执行→看报错→反思→重试，比Plan更适合迭代式修复。"

### Q3：为什么用BM25+FAISS而不是纯向量检索？
> "代码bug有精确关键词，'off_by_one'、'wrong_operator'是精准pattern。BM25处理精确匹配，FAISS处理语义相似，两者互补，RRF融合不依赖权重调参。"

### Q4：沙盒验证是离线还是实时？
> "推理阶段是实时验证：小模型生成→compile→测试用例执行→返回结果。训练阶段是离线验证：积累样本后离线计算reward。不要说反了。"

### Q5：reward hacking怎么防止？
> "三个机制：①六维互相制衡，单维hack其他维拖后腿；②KL散度约束限制策略偏离；③格式奖励前置——结构断裂直接0分，后续维度不再看。"

### Q6：为什么选GRPO而不是PPO/DPO？
> "DPO需要大量偏好对（几百上千条），187条数据直接过拟合——loss从800爆炸到40000。PPO需要额外Critic模型，显存要40GB，A100 32GB跑不起来。GRPO同组内互相做baseline，无需Critic，显存友好。"

### Q7：为什么是三轮ReAct？最多三轮的决策依据？
> "统计显示90%的可修复bug在前3轮内完成，第4轮及以后的修复成功率接近0。继续重试浪费算力。超过3轮的case存入L2反思记忆，为下次同类bug做准备。这是一个精度和效率的平衡点。"

### Q8：L2/L3记忆系统的价值是什么？
> "L2 Reflection让模型'记住失败的教训'，避免在同一类bug上重蹈覆辙；L3 Hermes Skill让模型'记住成功的套路'，下次遇到同类型直接复用SOP。两者都不占显存，只在推理时选择性注入相关条目，实现跨任务的经验积累。"
