## 一、项目介绍三版本（必须背熟）

### 15~20秒版（等电梯时间）
这是一个**多 Agent 协作的 AI 代码修复系统**，让 solver 生成、reviewer 验证、memory 提供经验、orchestrator 调度，形成闭环，把代码修复的 pass@1 从 54% 提到 87%。

### 30秒版（正式开场）
这是一个面向代码 Bug 修复的多 Agent 系统，解决的是单模型直接修代码容易成本高、不稳定、缺验证的问题。
我把它拆成四个角色：solver 负责生成修复、reviewer 负责编译测试和质量把关、memory 负责检索历史经验、orchestrator 负责流程控制和状态推进。核心思想是**生成和验证分离，经验和执行分离，调度和能力分离**，这样模块边界清晰，更容易做质量控制和扩展。

### 2分钟版（深入追问时）
我做的是一个多 Agent 的代码修复项目，叫 CodeFix-Agent。

**问题背景**：传统方式让一个模型直接修代码，容易修不对、成本高、没验证，模型可能"嘴上说修好了，测试一跑还是红的"。

**我的方案**：拆成多个明确角色：
- `orchestrator`：调度中枢，管任务生命周期、执行顺序、循环重试
- `solver`：按 CoT 结构生成修复代码
- `reviewer`：用编译、测试、Ruff 静态分析做客观验证
- `memory`：三层记忆（L1 工作记忆 / L2 失败教训 / L3 成功 SOP）+ 双池（E-pool 成功轨迹 / X-pool 候选策略）

**项目亮点**：compile-first 验证（编译器当 ground truth）、AST 精准定位（函数级字节偏移替换）、循环检测（4 维度 stall 检测）、安全沙箱（6 层隔离）、最小 A2A 受监督通信。

**核心指标**：pass@1 从 54% 提到 87%，大模型调用减少 60%，延迟降低 44%。

## 二、项目主线（必须背熟，一条线串起来）

整个项目讲成这条主线：

### 1. 问题是什么
- 单模型直接修：成本高、不稳定
- 没有验证闭环：模型说修好了，测试不一定过
- 没有记忆：每次都像第一次见到问题
- 边界情况容易漏

### 2. 我的方案
拆成四个角色各司其职：
- SolverAgent：专心生成
- ReviewerAgent：专心挑错
- MemoryAgent：专心提供上下文
- Orchestrator：专心控制流程

### 3. 为什么这么拆
"生成"和"验证"是两种不同能力。拆开以后更清晰：谁出问题就定位谁，更容易做重试和质量控制。

### 4. 流程怎么走
1. memory 检索历史经验
2. solver 生成第一版修复
3. reviewer 编译 + 测试 + 静态分析
4. 通过 → 结束；不通过 → 把反馈注入下一轮重试
5. 最多循环 3 轮，全程有循环检测兜底

### 5. 亮点是什么
不是用了多少名词，而是形成了完整闭环：**生成 → 验证 → 反馈 → 重试 → 记忆**，加安全沙箱做隔离。

## 三、每个模块怎么讲（高频追问应对）

### 3.1 Orchestrator（调度中枢）

**一句话**：它像项目经理，决定什么时候让哪个 agent 出场，自己不写代码也不存记忆。

**它做四件事**：
- 管任务生命周期（init → solving → reviewing → done/failed）
- 控执行顺序
- 管共享上下文（SharedContext）
- 控循环重试和终止条件

**追问：为什么不能省？**
因为多 agent 系统里必须有人统一管状态和流程。如果完全不要 orchestrator，agent 自己乱跑，执行顺序不可控、出错难定位、多轮重试容易死循环。我做的是"受监督的去中心化"，不是无政府状态。

**追问：你们用的是中心化还是去中心化？**
当前是 orchestrator-supervised 的半中心化。完全去中心化的问题在于：调试复杂度高、状态一致性难保证、消息风暴难以控制。所以我选了受监督的协作模式，既保留架构亮点，又可控可解释。

---

### 3.2 SolverAgent（生成者）

**一句话**：它按 CoT 结构生成修复代码，利用 memory 提供的经验做提示增强，但不负责最终验收。

**三个重点**：
- 首轮生成和重试生成逻辑不同（重试会注入审查反馈）
- 负责生成，不负责验收
- 通过 memory 检索相似经验做增强

**追问：为什么要单独拆 solver？**
因为生成修复方案本身是相对独立的认知任务。单独拆出来方便替换模型、改 prompt、做训练，职责边界也更容易维护。

---

### 3.3 ReviewerAgent（验证者）

**一句话**：它是系统里的"质检员"，用编译、测试、Ruff 静态分析做客观验证，把"语言上的正确"变成"执行上的正确"。

**它做什么**：
- 编译检查
- 测试执行（test runner）
- Ruff 静态分析（代码风格兜底）
- 7 维度 bug 检测
- Nudge 软验证

**追问：为什么不让 solver 自己检查？**
可以让 solver 自检，但不能只靠 solver 自检。生成者和审查者分离，能降低自我确认偏差，提高系统可靠性。这也是为什么 compile-first 很重要的原因——**编译器是接近 ground truth 的信号**，比模型自己的判断更可信。

---

### 3.4 MemoryAgent（记忆系统）

**一句话**：它把"过去做过的事"变成"这次可复用的经验"，不是简单存聊天记录，而是分层管理。

**三层记忆**：
- `L1`：当前任务上下文，类似工作记忆
- `L2`：失败经验，有时间衰减权重，类似踩坑记录
- `L3`：成功 SOP，类似解题模板

**双池设计**：
- `E-pool`（Exploitation Pool）：成功轨迹，偏稳定经验，命中重试成功率提升 18%
- `X-pool`（Exploration Pool）：候选策略，偏探索经验

**追问：memory 命中了和没命中怎么处理？**
命中了把相似经验作为提示增强；没命中就走冷启动流程，靠 solver 自己推理。memory 不强制依赖，它的价值在于减少重复试错。

---

### 3.5 SharedContext（共享上下文）

**一句话**：这次任务的"共享工作台"，所有 agent 不直接改彼此内部状态，而是通过 SharedContext 交换关键结果。

**里面存什么**：
- 原始 buggy code
- bug 描述 + test cases
- solver 输出 / reviewer 输出 / memory 输出
- 当前轮次、消息历史、轨迹数据

**为什么这样设计**：方便做状态追踪，也方便 orchestrator 统一管理。

### 4.2 Redis（即便项目没强依赖，也要能讲）

**面试官追问：如果系统走向生产，你觉得 Redis 适合放在哪些场景？**

你得按这个结构答：

#### 场景1：Session 缓存（最直接）

**项目现状**：session 存在内存 dict 里，重启就没了。

**如果用 Redis**：

```python
# 替代方案
import redis
r = redis.Redis(host='localhost', port=6379, db=0)

# Session 读写
r.setex(f"session:{session_id}", 3600, json.dumps(session_data))
data = r.get(f"session:{session_id}")
```

**为什么适合**：
- Session 本质是临时热点数据，读多写多，不需要强一致性
- Redis O(1) 读写，比文件 IO 快 10~100 倍
- 支持过期自动清理，省去手动管理
- 支持集群部署，多实例共享

**面试官追问：那为什么不一开始就用 Redis？**
因为项目阶段目标不同。原型阶段追求快速验证，引入 Redis 增加了运维复杂度（需要跑 Redis 服务）。等系统稳定了、需要多实例部署了，再迁移是合理的迭代节奏。

---

#### 场景2：API Key + 限流计数（项目里已经有了，但可以深化）

**项目现状**：APIKeyStore 用 Lock + 内存字典，单机没问题。

```python
# 当前实现（api_server.py）
def verify(self, key: str) -> Optional[dict]:
    with self._lock:  # 线程锁，只能单机有效
        entry["used_today"] += 1
        return dict(entry)
```

**如果用 Redis 做限流**：

```python
# 滑动窗口限流
import redis
r = redis.Redis()

def rate_limit(key: str, limit: int = 30, window: int = 60) -> bool:
    """
    限制每分钟最多 N 次请求
    用 Redis INCR + EXPIRE 实现滑动窗口
    """
    now = time.time()
    window_key = f"ratelimit:{key}:{int(now // window)}"
    count = r.incr(window_key)
    if count == 1:
        r.expire(window_key, window + 1)
    return count <= limit
```

**项目里已有**：
```python
# config.py
rate_limit_per_minute: int = 30
rate_limit_per_day: int = 1000
```

**面试官追问：分布式环境下怎么限流？**
单机用锁就行，但多实例部署时必须用 Redis 作为共享计数器。每个实例把请求量写到 Redis 的同一个 key，INCR 原子操作保证计数准确，EXPIRE 自动清理过期记录防止内存泄漏。

#### 场景3：Memory 热数据缓存（加速 RAG 检索）

**项目现状**：E-pool 和 X-pool 存 JSON 文件，每次检索要读文件、解析 JSON。

**如果用 Redis**：

```python
# 热门记忆碎片缓存到 Redis
r.hset("memory:e_pool", piece_id, json.dumps(piece_data))
r.expire("memory:e_pool", 3600)  # 1小时过期

# 命中时优先从 Redis 读，miss 了再读文件 + 回填缓存
cached = r.hget("memory:e_pool", piece_id)
if cached:
    return json.loads(cached)
```

**为什么适合**：RAG 检索时频繁读 E-pool，热点数据放 Redis 能显著降低 IO 开销。

#### 场景4：分布式锁（多 Agent 协作时的资源竞争）

**面试官追问：你们多 Agent 协作，如果有竞争同一个资源的场景怎么办？**

**项目现状**：多进程写 `shared_e_pool.jsonl` 时用文件锁，但比较简陋：

```python
# multi_agent_orchestrator.py 里的简单文件锁
def _acquire_lock(self) -> bool:
    for _ in range(10):
        try:
            fd = os.open(str(self._lock_file), os.O_CREAT | os.O_EXCL | os.O_WRONLY)
            os.close(fd)
            return True
        except FileExistsError:
            time.sleep(0.1)
    return False
```

**如果用 Redis 分布式锁**：

```python
import redis

def acquire_lock(lock_name: str, timeout: int = 10) -> bool:
    """
    SET NX EX 模式：原子性获取锁
    """
    lock_key = f"lock:{lock_name}"
    acquired = r.set(lock_key, "1", nx=True, ex=timeout)
    return bool(acquired)

def release_lock(lock_name: str):
    r.delete(f"lock:{lock_name}")

# 使用场景：多 Agent 同时写记忆池时
if acquire_lock("memory_write"):
    try:
        # 安全的写操作
        memory_agent.write_to_pool(context)
    finally:
        release_lock("memory_write")
```

**为什么适合**：
- Redis 分布式锁不依赖进程/机器，是跨实例的
- SET NX EX 是原子操作，不会出现锁竞争漏洞
- 比文件锁可靠，支持自动过期防止死锁

**面试官追问：Redis 锁有什么坑？**
两个经典坑：
1. **锁过期了任务还没跑完**——解决：看门狗续期机制（RedLock 算法）
2. **主从切换丢锁**——解决：用 RedLock 多节点多数投票

#### 场景5：消息队列削峰（未来扩展）

**面试官追问：这个系统如果并发量上来，你觉得哪里需要 MQ？**

**适合放 MQ 的链路**：

```
1. 代码修复请求 → 接入层
   用户请求先写入 MQ，API 层快速响应，后台 worker 慢慢处理
   这样 API 不被长时间占用，支持高并发

2. sandbox 执行结果回调
   代码执行可能耗时长，放 MQ 解耦，避免阻塞主流程

3. 多 Agent 之间的异步消息
   solver → reviewer → solver 的循环可以异步化
   reviewer 发现问题，往 MQ 发消息，solver 订阅消费
```

**MQ 选型建议**（如果被追问）：
- **轻量场景 / 单机**：Redis Stream（项目里已经有 Redis 的话，不用额外引入 Kafka）
- **生产环境 / 高吞吐**：RabbitMQ 或 Kafka
- **有延迟任务需求**：Redis + 定时任务，或 Celery（基于 MQ 的 Python 任务队列）

**为什么不是现在引入**：MQ 增加了系统复杂度（运维、消息幂等、顺序保证），当前 QPS 不高，加 MQ 反而过度设计。

#### 事务怎么用

**场景：用户请求 + 配额扣减必须原子**

```python
# 用 MySQL 事务保证一致性
import pymysql

def handle_fix_request(api_key: str, request_data: dict):
    conn = pymysql.connect(host='localhost', user='root', password='xxx', database='codefix')
    try:
        with conn.cursor() as cursor:
            # 开启事务
            conn.begin()

            # 1. 检查并扣配额（原子操作）
            affected = cursor.execute("""
                UPDATE api_keys
                SET used_today = used_today + 1
                WHERE key_hash = %s
                  AND used_today < quota_per_day
                  AND revoked_at IS NULL
            """, (hash_key(api_key),))

            if affected == 0:
                raise Exception("配额不足或Key无效")

            # 2. 写入请求记录
            cursor.execute("""
                INSERT INTO requests (api_key_id, request_data, status)
                VALUES (%s, %s, 'processing')
            """, (api_key_id, json.dumps(request_data)))

            conn.commit()
    except Exception as e:
        conn.rollback()
        raise e
    finally:
        conn.close()
```

#### 索引怎么建

**常见慢查询场景**：

```sql
-- 按用户查活跃会话（需要索引）
CREATE INDEX idx_user_status ON sessions(user_id, status);

-- 按任务类型查记忆碎片（需要索引）
CREATE INDEX idx_task_type ON memory_pieces(task_type);

-- 查某段时间内的高质量记忆（复合索引）
CREATE INDEX idx_quality_time ON memory_pieces(
    judge_score_avg DESC,
    execution_pass_rate DESC,
    created_at DESC
);
```

**面试官追问：什么时候建索引？**
- WHERE 条件里经常出现的字段
- JOIN 的关联字段
- ORDER BY 经常出现的字段
- 不要过度建索引：每次 INSERT/UPDATE 都要维护，多了影响写入性能

#### 隔离级别怎么选

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 适合场景 |
|---------|-----|----------|-----|---------|
| READ UNCOMMITTED | 可能 | 可能 | 可能 | 基本不用 |
| READ COMMITTED | 不可能 | 可能 | 可能 | 大多数场景 |
| REPEATABLE READ（MySQL默认） | 不可能 | 不可能 | 可能 | 需要强一致性 |
| SERIALIZABLE | 不可能 | 不可能 | 不可能 | 极端场景，性能差 |

**项目场景用哪个**：
- **配额扣减**：需要 `REPEATABLE READ` + 行锁，防止同一 Key 并发请求都读到相同配额然后都扣成功
- **会话读取**：可以 `READ COMMITTED`，读到的稍微旧一点无所谓
- **记忆碎片写入**：可以 `READ COMMITTED`，记忆系统本身就有重试机制

```sql
-- 配额扣减的锁写法
SELECT id, quota_per_day, used_today FROM api_keys
WHERE key_hash = ? FOR UPDATE;  -- FOR UPDATE 加行锁
```

#### 慢查询怎么排查

**三板斧**：

```sql
-- 1. 开启慢查询日志
SET GLOBAL slow_query_log = 'ON';
SET GLOBAL long_query_time = 1;  -- 超过1秒记录

-- 2. 查看慢查询
SHOW VARIABLES LIKE 'slow_query_log_file';
-- 然后 tail -f 查看日志

-- 3. 用 EXPLAIN 分析
EXPLAIN SELECT * FROM sessions
WHERE user_id = 123 AND status = 'active'
ORDER BY created_at DESC LIMIT 20;
```

**常见优化手段**：
- 索引没建 → 加索引
- 全表扫描 → 看执行计划，调整查询条件
- SELECT * → 只查需要的字段
- 大量 INSERT → 批量插入 + 关闭索引（先删后建）
- 深分页 → 用游标或延迟关联，不用 `LIMIT 10000, 20`

### 4.4 消息队列（即便项目未深度落地，也要能讲）

**面试官追问：你们这个系统未来适合引入 MQ 吗？适合放在哪些链路？**

#### 为什么这个系统未来可能需要 MQ

**当前系统的瓶颈**：
- `/fix` 接口是同步的，用户要等 solver + reviewer 全跑完才能拿到结果
- 如果并发量上来，sandbox 执行慢会拖垮 API 响应时间
- 多 Agent 循环重试时，reviewer 反馈 → solver 重试是同步串行的

#### 适合放 MQ 的三个链路

**链路1：代码修复请求（削峰填谷）**

```
用户请求 → API 写入 MQ → 立即返回 202 Accepted → 后台 worker 消费 MQ → 执行修复 → 结果落库 → 回调通知
```

- API 层不用等代码跑完，快速响应
- 请求高峰时 MQ 缓冲，防止系统过载
- worker 可以弹性扩缩容

**链路2：Sandbox 执行结果异步回调**

```
sandbox 执行时间长（可达几十秒），如果同步等待：
- 用户体验差（超时）
- API 连接占用高

改成：
提交执行任务 → MQ 发消息 → 立即返回 task_id → 轮询 / WebSocket 拿结果
```

**链路3：多 Agent 异步协作**

```
当前：solver → reviewer → (阻塞等待) → solver 重试

未来：
  solver 完成任务 → 发 MQ 消息 → reviewer 消费 → 审查完再发 MQ → solver 消费重试

好处：
- solver 和 reviewer 可以并行各自跑
- 系统吞吐更高
- 适合更大规模的 Agent 协作
```

#### MQ 选型对比

| 特性 | RabbitMQ | Kafka | Redis Stream |
|------|---------|-------|-------------|
| 吞吐量 | 中（万级/秒）| 高（百万级/秒）| 中（万级/秒）|
| 延迟 | 低 | 低 | 极低 |
| 持久化 | 支持 | 支持 | 支持 |
| 消息回溯 | 不支持 | 支持（offset）| 不支持 |
| 运维复杂度 | 中 | 高 | 低（项目已有Redis）|
| 适合场景 | 可靠传递 | 大数据流处理 | 轻量异步任务 |

**我的选择建议**：
- **如果当前阶段引入**：用 Redis Stream（复用现有 Redis，不用额外运维）
- **如果未来高吞吐**：迁移到 Kafka（需要额外运维，但吞吐更强）
- **如果需要消息回溯重放**：用 Kafka（训练数据回放场景很适合）

#### Redis Stream 实现轻量 MQ

```python
import redis

r = redis.Redis()

# 生产者：提交任务
task_id = r.xadd("fix_tasks", {
    "buggy_code": buggy_code,
    "bug_description": bug_description,
    "priority": "normal"
})
return {"task_id": task_id, "status": "queued"}

# 消费者：后台 worker
while True:
    # 阻塞读取，最多等 5 秒
    messages = r.xreadgroup(
        "fix_workers", "worker_1",
        {"fix_tasks": ">"},  # 只读新消息
        count=1, block=5000
    )
    for stream, msgs in messages:
        for msg_id, data in msgs:
            # 处理任务
            result = process_fix(data)
            # 更新状态
            r.xadd("fix_results", {
                "task_id": msg_id,
                "status": "done",
                "result": json.dumps(result)
            })
            # 确认已处理
            r.xack("fix_tasks", "fix_workers", msg_id)
```

**面试官追问：MQ 挂了怎么办？**
- 消息持久化到磁盘，Redis RDB/AOF 保障
- 生产者发送时 `PUBLISH` 或 `XADD`（持久化消息）
- 消费者用 `xreadgroup` + `xack`，消息处理完才确认，没确认的重启后会重新投递
- 幂等处理：消费前查重（task_id 去重），防止重复执行

**面试官追问：什么时候用线程锁，什么时候用异步锁？**

#### GIL（被追问时别说错）

**项目背景**：大量 Python 代码，为什么性能还能做到 pass@1 提升 33%？

**你得会讲**：
- Python GIL 导致同一时刻只有一个线程执行 Python 字节码
- 但项目里不靠多线程做计算密集型加速，而是：
  - **IO 密集型**：调用 Ollama API、读写文件——GIL 在 IO 等待时会释放，多线程有效

**如果被追问更深入**：
"我不认为 GIL 是这个项目的瓶颈。系统的主要延迟来自模型推理和网络 IO，不是 Python 层面的计算。如果未来需要更大规模并行，可以用多进程 + Redis 共享状态，或者直接上 Ray 这样的分布式框架。"

## 五、A2A 怎么讲（项目亮点，高频追问）

**先说结论**：我做的是最小可控、受监督的 A2A，不是完全去中心化。

### 为什么不是完全 A2A
因为完全 A2A 的问题：调试复杂度高、权限控制难、容易消息风暴、状态一致性难保证。

### 当前最小 A2A 允许哪些流
就两条：
- `solver → memory`：solver 主动查询相似经验
- `reviewer → solver`：reviewer 把审查反馈直接路由给 solver

### 追问：怎么防止 agent 乱发消息？
路由注册 + 白名单权限控制，只允许显式声明的通信路径；消息记录进 message history，便于审计和循环检测。

### 追问：A2A 带来了什么收益？
架构表达更完整（从纯 orchestrator 协调演进到受监督协作），后续扩展更方便（增加 planner、validator 时可复用同一消息机制）。

### 6.1 Compile-First

## 八、必答题（8 道，背熟）

### 1. 为什么要多 Agent？
代码修复包含生成、验证、记忆复用、流程控制等多类能力，全塞进一个 agent 职责过重、错误定位困难。拆开后每个模块边界清晰，更容易做闭环验证和重试。

### 2. 为什么有 reviewer？
"模型说修好了"和"代码真的修好了"是两回事。reviewer 用编译、测试和静态分析把关，降低幻觉和误修风险。

### 3. 为什么有 memory？
很多 bug 模式可复用。memory 把过去失败经验和成功 SOP 沉淀下来，减少重复试错，L2 命中后重试成功率提升 18%。

### 4. orchestrator 为什么不能省？
它负责统一状态、执行顺序、异常处理和重试终止条件。完全不要 orchestrator，系统复杂度、调试成本和失控风险都会上升。

### 5. 为什么 A2A 只做最小版？
工程上追求收益和成本平衡。最小版能展示架构演进方向，又不引入大规模消息治理问题。

### 6. 项目最大不足是什么？
三方面：复杂任务下整体稳定性还需更多评测验证；memory 命中质量和收益还需更细量化分析；A2A 后续扩展需要更完整的权限和回路控制机制。这个回答既承认不足，又显得你知道下一步该做什么。

### 7. 如果并发量上来，系统哪里是瓶颈？
三个潜在瓶颈：1）sandbox 执行是同步的，高并发会排队；2）memory 检索依赖文件 IO，QPS 高了会慢；3）orchestrator 是单进程调度，多实例部署需要共享状态。解法分别是引入 MQ、做 Redis 缓存、加 Redis 分布式协调。

### 8. 为什么用 FastAPI 而不是 Django/Flask？
FastAPI 是异步的，性能高，类型提示完整（自动生成 OpenAPI 文档），依赖注入适合做认证和限流这些横切关注点。Django 太重，Flask 是同步的。
