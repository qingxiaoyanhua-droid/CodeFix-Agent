## 二、核心知识点深度问答

### Q1: 为什么选择 ReAct 架构而不是 Plan-and-Execute 或 Toolformer？

**参考答案：**

ReAct（Reasoning + Acting）的核心思想是在每个推理步骤中交替进行**思考**和**工具调用**，让模型能够自我纠错。我的项目中：

- `cot_react_agent.py` 实现了小模型生成四步 CoT 推理 → 编译验证 → 失败则大模型反馈
- **为什么不用 Plan-and-Execute？** 因为规划-执行模式需要在执行前生成完整计划，但代码修复是高度迭代的过程，编译器和测试用例提供了 Ground Truth 的外部反馈，Plan-and-Execute 的静态规划无法利用这种动态验证信号
- **为什么不用 Toolformer？** Toolformer 专注于让模型学会调用工具，但缺乏显式的推理链，代码修复需要理解 bug 的因果链条（off-by-one、逻辑错误等），CoT 推理是核心

ReAct 的关键优势：编译器 = 免费验证器，AST diff = 可解释的中间状态，CoT 让每一步失败都有迹可循。

**追问1：ReAct 和 Reflexion 的区别？**
Reflexion 在 ReAct 基础上增加了语言强化反馈，Agent 输出执行轨迹 → 外部评估器打分 → Agent 根据反馈反思。我的项目中 L2 反思记忆（`reflection_memory`）其实就是 Reflexion 思想，但通过结构化的 `Reflection` 数据结构实现，而非纯自然语言反馈。

**追问2：CoT 的 4 步具体是哪 4 步？**
按照 `dual_pool_memory.py` 的 stage_pool 设计（L1-L4）：
1. **L1-定位**：识别 bug 类型和影响函数
2. **L2-分析**：定位错误根因（off-by-one/null-pointer/逻辑错误）
3. **L3-推理**：结合 RAG 案例和记忆生成修复策略
4. **L4-验证**：编译 + 测试用例确认

### Q2: 小模型生成 + 大模型验证的动机？trade-off 分析？

**参考答案：**

**动机（三层递进）：**

1. **成本驱动**：Qwen2.5-Coder-1.5B 推理成本约为 Qwen2.5-Coder-7B 的 1/5。项目实测大模型调用减少 60%，延迟降低 44%。

2. **速度驱动**：1.5B 模型在消费级 GPU 即可快速推理，避免大模型等待时间。

3. **任务特性驱动**：代码修复的"定位"和"简单修复"（语法、简单逻辑）可以被小模型搞定，只有复杂 case 才需要大模型的语义理解能力。`task_planner.py` 的 `complexity_estimator` 正是做这个路由决策的——复杂度 >= COMPLEX 才触发大模型。

**Trade-off：**

| 维度 | 小模型 | 大模型 |
|------|--------|--------|
| 推理能力 | 简单 bug 修复 ok | 复杂语义理解强 |
| 幻觉风险 | 较高 | 较低 |
| 延迟 | ~100ms | ~1-2s |
| 成本 | 低 | 高 |
| CoT 质量 | 基础 | 深入 |

**面试官追问：条件升级的触发条件是什么？**

看 `task_planner.py` 第 505-514 行和 `cot_react_agent.py` 的逻辑：
- 编译失败 → 触发大模型错误分析
- `complexity >= COMPLEX` → 概率性触发大模型（概率 = 复杂度值/5）
- `intent == REPO_LEVEL` → 强制大模型协调

### Q3: AST 动态路由怎么实现的？行号偏移问题怎么解决？

**参考答案：**

**AST 精确定位**（`task_planner.py` 第 426-436 行）：
```python
steps.append(PlanStep(
    name="AST_函数定位",
    tool="ast_locate",
    input_requirements={"code": buggy_code, "functions": intent.affected_functions},
    expected_output="FunctionInfo list with byte offsets",  # 注意是字节偏移，不是行号！
))
```

**为什么用字节偏移而非行号？**
当修改代码时（增删行），所有后续行的行号都会偏移。Python AST 的 `node.col_offset` 和 `end_col_offset` 是基于字符位置的，即使代码被修改，字节偏移仍可通过相对位置计算新位置。

**AST 动态路由的核心流程**（`cot_react_agent.py`）：
1. `IntentClassifier` 用正则检测 bug 类型 → 提取 `affected_functions`
2. AST 解析代码，定位到具体 `FunctionDef` 节点 → 记录 `lineno` 和 `col_offset`
3. 根据 bug 类型选择工具链：
   - `SYNTAX_FIX` → 直接修复，不用 CoT
   - `CRASH_FIX` → 先 `exec()` 捕获错误栈，再 CoT
   - `BUG_FIX` → 完整 CoT + 编译验证
   - `REPO_LEVEL` → grep + AST 跨文件定位

**追问：AST 定位在多语言场景怎么处理？**

沙箱中支持 Python/Node/C/C++。Python 直接用标准库 `ast`；其他语言需要外部工具（Tree-sitter），或者对于编译型语言，编译器报错本身就包含了精确的行号信息。

### Q4: 三层记忆系统设计？L1/L2/L3 具体存什么？

**参考答案：**

基于 `cot_react_agent.py` 和 `dual_pool_memory.py` 的实现：

**L1 — 会话上下文记忆（AST diff 压缩）**
- 存储：当前 session 内的 AST diff（函数级别的修改记录）
- 目的：在同一个 bug 修复的多次迭代中，避免重复推理
- 实现：压缩为"修复前函数签名 + 修复后函数签名"的 diff 快照
- `cot_react_agent.py` 中的 `compact_ast_diff()` 即做这件事

**L2 — 反思记忆（失败经验结构化）**
- 存储：`Reflection` 对象，包含 `bug_pattern`（如"off-by-one in range() loop"）+ `root_cause` + `fix_strategy`
- 来源：每次修复失败后，Agent 反思失败原因，写入 L2
- 检索：当新任务的 bug 类型匹配时，从 L2 检索相似失败经验
- 质量门槛：只有 `verification_count >= 2` 且 `helpfulness_score >= 0.6` 的反射才会被用于检索

**L3 — 技能记忆（成功 SOP 提取）**
- 存储：`Skill` 对象，包含 `pattern` + `standard_operation_procedure`（标准操作流程）
- 来源：每次成功修复后，提取通用的修复模式（如"修复索引越界 → 检查边界条件"）
- L3 技能是跨 bug 类型的通用知识，可被重复使用

**三层协作流：**
```
新任务 → L1检查(同会话去重) → L2检查(失败教训) → L3检查(成功SOP) → RAG检索 → 小模型推理
```

**追问：记忆的脏数据问题怎么解决？**

`dual_pool_memory.py` 第 406-421 行的 `prune_low_quality()` 机制：
- 定期清理：`quality_threshold = 0.3` 以下的碎片被删除
- 质量门槛：写入时要求 `quality_score() >= 0.6`（综合 LLM-judge 评分和测试通过率）
- 时间衰减：`relevance_score()` 中有 7 天半衰期的衰减因子
- 动态路由权重：`w_E` 和 `w_X` 由 LLM-as-a-Judge 反馈动态调整

### Q5: BM25 + FAISS 混合检索为什么有效？两者互补性在哪里？

**参考答案：**

**BM25 的优势与局限：**
- 基于词频-逆文档频率，擅长精确关键词匹配（如 bug 类型名 "off-by-one"）
- `enterprise_rag_pipeline.py` 中 BM25 实现：`k1=1.5, b=0.75`，可调参数
- 局限：无法捕捉语义相似性，"数组越界"和"索引错误"用 BM25 可能匹配不到

**FAISS（向量检索）的优势与局限：**
- 基于语义嵌入，"数组越界"和"索引错误"都能检索到相关内容
- `EmbeddingModel` 使用 `all-MiniLM-L6-v2`（384维向量）
- 局限：对专有名词（如特定 API 函数名）有时不如 BM25 精确

**混合检索的实现**（`enterprise_rag_pipeline.py` 第 707-733 行）：
```python
# 两个检索分别跑 top_k*2 的候选
bm25_results = self.bm25_index.search(query, top_k=top_k * 2)
query_embedding = self.embedding_model.encode_query(query)
vector_scores = self._cosine_similarity(query_embedding, self.chunk_embeddings)

# 归一化后加权融合
combined_scores[i] = bm25_weight * bm25_score + vector_weight * vector_score
# bm25_weight=0.4, vector_weight=0.4
```

**为什么用 0.4/0.4 而不是其他权重？**
实际项目中通过网格搜索确定权重。0.4/0.4 是一个相对平衡的配置——BM25 保证关键词命中，向量检索补充语义相似性。权重可调，`config.py` 中 `rag.bm25_weight=0.6, rag.vector_weight=0.4`（实际生产配置）。

**追问：重排序模型的作用？**

粗筛阶段用 BM25+向量选出 top_k*2 的候选（可能几十条），然后用 `cross-encoder/ms-marco-MiniLM-L-6-v2` 对 query-chunk 对做精细相关性打分，只返回 top_k 条。`Reranker` 类实现了这个两阶段流程。

### Q6: GRPO 相比 PPO 的优势？为什么选 GRPO？

**参考答案：**

**GRPO（Group Relative Policy Optimization）核心思想：**

DeepSeekMath 论文提出，针对有参考答案的任务，不需要单独的价值网络，而是：
1. 对同一个 prompt 生成 G=8 个候选回复（`num_generations=8`）
2. 用 reward model 给每个候选打分
3. 相对于组内平均 reward 更新策略

**相比 PPO 的优势：**

| 维度 | PPO | GRPO |
|------|-----|------|
| 价值网络 | 需要单独训练 | 不需要，节省 50% 显存 |
| 样本效率 | 较低（on-policy） | 较高（组内相对比较） |
| 超参数 | 大量（clip、entropy系数等） | 少（主要调 group size 和 reward） |
| 适合场景 | 开放生成 | 有明确 reward 的任务（代码修复完美匹配） |

**代码实现**（`llm_critic_reward.py`）：
```python
# GRPO 的 reward_function 综合了多粒度奖励：
score = format_reward(0.3) + syntax_reward(0.3) + llm_critic_reward(0.4)
```

**追问：多粒度奖励如何防止 Reward Hacking？**

Reward Hacking 是模型找到"骗过 reward 但不真正解决问题"的漏洞。多粒度设计让漏洞空间极小：
- 格式奖励（0.3）：必须有 `<reasoning>` 和 `<fixed_code>` 标签
- 语法奖励（0.3）：必须通过 `compile()` 检查
- AST 结构匹配（测试框架提供）：测试用例执行必须全部通过
- LLM Critic（0.4）：大模型评估补丁的正确性、质量、简洁性，捕捉"编译通过但逻辑错误"的情况

每一层奖励都需要不同维度的能力才能获得，模型很难同时在所有维度作弊。

### Q7: 手写 RMSNorm / RoPE / GQA 实现时踩过哪些坑？

**参考答案：**

**RMSNorm（均方根归一化）：**
```python
def rms_norm(x, weight, eps=1e-6):
    # 踩坑1：不要用 mean(x^2)，要用 rms = sqrt(mean(x^2))
    # 踩坑2：weight 是可学习的 scale 参数，不能漏掉
    rms = (x.pow(2).mean(-1, keepdim=True) + eps).rsqrt()  # rsqrt = 1/sqrt
    return weight * x * rms
```
- 相比 LayerNorm：不需要计算均值，只用 RMS，节省了约 50% 的计算量
- 坑：rsqrt 的数值稳定性，eps 要足够大防止除零

**RoPE（旋转位置编码）：**
```python
def apply_rope(x, freqs_cis):
    # x: [batch, seq, head, dim]
    # freqs_cis: 预计算的复数旋转因子 [seq, dim//2]
    x_complex = x[..., :x.shape[-1]//2].unsqueeze(-1) * freqs_cis.unsqueeze(0).unsqueeze(0)
    x_out = torch.cat([x_complex.real - x_complex.imag, x_complex.real + x_complex.imag], dim=-1)
```
- 核心思想：旋转 query 和 key 的嵌入，让相对位置通过复数乘法编码
- 踩坑：dim 必须是偶数；旋转角度 $\theta = 10000^{2i/dim}$ 需要预计算
- 相比绝对位置编码的优势：天然支持任意长度的上下文外推

**GQA（分组查询注意力）：**
```python
# 标准 MHA: 每个 head 都有独立的 Q, K, V → 计算量 O(n*d^2)
# GQA: G 个 query heads 共享 1 组 K, V heads → 计算量降低 G 倍
# MQA: 所有 query heads 共享 1 组 K, V → 极端压缩

# 实现：创建 num_q_heads 个 query projection，但只创建 num_kv_heads 个 k/v projection
q_proj = nn.Linear(d_model, num_q_heads * head_dim)
k_proj = nn.Linear(d_model, num_kv_heads * head_dim)  # num_kv_heads << num_q_heads
v_proj = nn.Linear(d_model, num_kv_heads * head_dim)
```
- 踩坑：attention 计算时需要把 Q 按照 group 展开以便和 K/V 对齐
- 8 层轻量化模型用 GQA，在 1.5B 规模下将 KV cache 减少约 3-4 倍

### Q9: FastAPI 生产部署的关键设计（鉴权、限流、审计）？

**参考答案：**

**API Key 鉴权**（`api_server.py` 第 78-176 行）：
```python
class APIKeyStore:
    # JSON 持久化，格式: {"cfx_<hex>": {"owner": "...", "quota_per_day": 1000, ...}}
    def verify(self, key: str) -> Optional[dict]:
        self._check_reset()  # 每日重置配额
        entry = self._keys.get(key)
        if entry["used_today"] >= entry["quota_per_day"]:
            return None  # 配额耗尽
        entry["used_today"] += 1
        return entry
```
- Key 格式：`cfx_` + 32 位随机 hex
- 每日配额重置（`_reset()` 在 `_check_reset()` 中触发）

**滑动窗口限流**（第 208-254 行）：
```python
class SlidingWindowRateLimiter:
    # 每个 key 维护一个 deque of timestamps
    def check(self, key: str) -> tuple[bool, int]:
        self._evict_old(bucket)  # 踢掉 60s 外的请求
        if len(bucket) >= self.max_requests:  # 30 req
            return False, 0
        bucket.append(time.time())
```

### Q10: 编译优先（Compile-First）如何实现零幻觉？

**参考答案：**

幻觉（hallucination）= 模型生成看起来合理但实际错误的代码。在代码修复场景中：
- **传统方案**：全量依赖大模型生成代码 → 幻觉率高（可能生成语法正确但逻辑错误的代码）
- **Compile-First**：用确定性工具（编译器、测试用例）作为 Ground Truth，将"幻觉"问题转化为"工具调用"问题

**实现**（`cot_react_agent.py` 和 `task_planner.py`）：
```
小模型生成修复代码
    ↓
立即编译验证（compile() 或 gcc）
    ↓ pass → 测试用例执行 → 通过则输出
    ↓ fail → 大模型分析错误（hint_only，不亲自修）
```

**零幻觉的三道防线：**
1. **语法层**（Layer 1）：`compile()` 检查语法，任何语法错误立即重试
2. **类型层**（Layer 2）：Python 是动态类型，但 Ruff 静态分析检查类型一致性
3. **语义层**（Layer 3）：测试用例执行，确认修复后行为符合预期

编译器是确定性的，没有幻觉。测试用例也是确定性的。"零幻觉"不是说模型不出错，而是模型的输出会经过确定性工具的严格验证，任何错误都会被拦截。

- 逻辑正确但不够简洁（LLM critic 捕捉）
- 测试用例全通过（测试执行奖励）

### 1. BM25 实现
```python
import math
from collections import Counter

def tokenize(text: str) -> list[str]:
    return re.findall(r'\b\w+\b', text.lower())

def bm25_score(query: str, doc: str, doc_freq: dict, N: int, avgdl: float, k1=1.5, b=0.75) -> float:
    tokens = tokenize(doc)
    doc_len = len(tokens)
    tf = Counter(tokens)
    score = 0.0
    for term in tokenize(query):
        if term in doc_freq:
            idf = math.log((N - doc_freq[term] + 0.5) / (doc_freq[term] + 0.5) + 1)
            qtf = tf.get(term, 0)
            numerator = qtf * (k1 + 1)
            denominator = qtf + k1 * (1 - b + b * doc_len / avgdl)
            score += idf * numerator / denominator
    return score
```

### 2. 实现 RMSNorm
```python
import torch
import torch.nn as nn

class RMSNorm(nn.Module):
    def __init__(self, d_model: int, eps: float = 1e-6):
        super().__init__()
        self.eps = eps
        self.weight = nn.Parameter(torch.ones(d_model))

    def forward(self, x: torch.Tensor) -> torch.Tensor:
        # x: [..., d_model]
        rms = x.pow(2).mean(-1, keepdim=True).add(self.eps).rsqrt()
        return self.weight * x * rms
```

### 3. AST 函数级代码替换（解决行号偏移）
```python
import ast

def replace_function_by_name(code: str, func_name: str, new_body: str) -> str:
    """
    基于函数名精确定位并替换函数体。
    解决：直接在 source 上用行号替换会因缩进/换行导致偏移。
    解决：利用 AST 的字节偏移，转换为字符位置后精确替换。
    """
    tree = ast.parse(code)
    target_node = None
    for node in ast.walk(tree):
        if isinstance(node, ast.FunctionDef) and node.name == func_name:
            target_node = node
            break

    if target_node is None:
        raise ValueError(f"Function '{func_name}' not found")

    # AST 提供精确的字节偏移（col_offset 是字符偏移，不是字节偏移）
    lines = code.splitlines(keepends=True)
    start_line = target_node.lineno - 1  # 0-indexed
    end_line = target_node.end_lineno     # inclusive, 0-indexed

    # 计算缩进
    first_line = lines[start_line]
    indent = len(first_line) - len(first_line.lstrip())
    indent_str = first_line[:indent]

    # 构造新函数体
    new_body_lines = new_body.strip().splitlines()
    new_body_indented = '\n'.join(indent_str + '    ' + l if l.strip() else l
                                    for l in new_body_lines)

    # 替换
    new_lines = lines[:start_line]
    new_lines.append(indent_str + f"def {func_name}():\n")
    new_lines.append(new_body_indented + '\n')
    new_lines.extend(lines[end_line + 1:])

    return ''.join(new_lines)
```

### 4. 滑动窗口限流器
```python
from collections import defaultdict, deque
import time

class SlidingWindowRateLimiter:
    def __init__(self, max_requests: int, window_seconds: int):
        self.max_requests = max_requests
        self.window_seconds = window_seconds
        self._buckets: dict[str, deque] = defaultdict(lambda: deque(maxlen=max_requests + 1))

    def check(self, key: str) -> bool:
        now = time.time()
        cutoff = now - self.window_seconds
        bucket = self._buckets[key]

        # 驱逐过期请求
        while bucket and bucket[0] < cutoff:
            bucket.popleft()

        if len(bucket) >= self.max_requests:
            return False

        bucket.append(now)
        return True
```

### 5. DualPool 路由器
```python
import numpy as np

class Router:
    """
    E-pool (Exploitation): 存储成功经验
    X-pool (Exploration):  存储新候选
    路由器决策: 以概率 α = w_E / (w_E + w_X) 选择 E-pool
    """
    def __init__(self, w_E=1.0, w_X=1.0):
        self.w_E = w_E
        self.w_X = w_X

    def decide(self, task: dict) -> str:
        # 如果 E-pool 有高相关记忆 → 强制用 E
        if self._has_high_relevant(task, pool="E"):
            return "E"

        # 否则按概率选择
        alpha = self.w_E / (self.w_E + self.w_X)
        if np.random.random() < alpha:
            return "E"
        else:
            return "X"

    def update(self, used_pool: str, delta: float):
        """
        delta: judge 评估结果 (1=成功, 0=失败)
        成功时增强该池权重，失败时衰减
        """
        if used_pool == "E":
            self.w_E = self.w_E + 0.5 if delta > 0 else max(1.0, 0.5 * self.w_E)
        else:
            self.w_E = max(1.0, 0.5 * self.w_E) if delta > 0 else self.w_E + 0.5
```

## 四、开放性问题

### Q1: 如果重新做这个项目，会有什么不同的设计决策？

1. **小模型换成更小的**：1.5B 对简单 bug 仍然过大了，考虑用 0.5B 或 CodeGen-Multi 的 350M 做极轻量推理，1.5B 只做复杂 case
2. **RAG 从 BM25+FAISS 升级**：考虑用 ColBERTv2 或混合检索的 late-interaction 模式，提升检索精度
3. **评估从 pass@1 升级**：增加修复后代码的 LLM-as-Judge 评分作为质量维度
4. **记忆系统引入向量数据库**：从 JSON 文件升级到 Qdrant/Milvus，支持分布式检索

### Q2: 如何 scale 到更大的代码库（多文件、跨模块）？

当前 REPO_LEVEL 意图处理比较粗糙（grep + AST 定位）。scalable 的方案：
1. **代码依赖图**：用 AST 构建 import 依赖图，从 bug 文件向外辐射定位相关文件
2. **分片处理**：按模块分片，并行处理不同模块的 bug，最后合并
3. **大模型协调**：用大模型做仓库级理解（函数调用关系、接口契约），小模型做具体修改

### Q3: GRPO 训练崩溃了怎么办？

可能原因：
1. **Reward Hacking**：多粒度奖励仍有漏洞，模型找到了作弊路径
2. **KL 散度爆炸**：策略更新步长过大导致崩溃
3. **奖励信号稀疏**：某些 bug 类型难以获得正向奖励

解决方案：
1. 增大 group size（8→16）增加正样本密度
2. 加入 KL 惩罚项限制策略更新幅度
3. 课程学习：从简单 bug 开始训练，逐步引入复杂 bug

### Q4: 模型在实际部署中遇到过什么意想不到的 bug？

**案例1**：模型学会了"通过修改测试用例"来通过测试，而不是修改代码。解决方案：测试用例必须从外部注入，不在模型可修改范围内。

**案例2**：Ruff 兜底机制反而导致了新的 bug——Ruff 的"自动优化"有时改变了代码语义。解决方案：Ruff 只做 style check，不做 auto-fix。

### Q5: 项目的技术护城河是什么？

1. **三层记忆 + RAG 的协同**：不是简单的向量检索，而是 L1/L2/L3 + 知识库的协同，不同层级解决不同粒度的问题
2. **编译优先的架构设计**：将确定性工具（编译器、测试）作为 Ground Truth，从根本上消灭幻觉
3. **GRPO 多粒度奖励**：5 维奖励设计（格式+语法+AST+测试+LLM Critic）提供密集训练信号，是竞品难以复制的工程壁垒
