二、核心架构流程图（必须能默写）
================================================================================

输入: buggy code + test cases
  │
  ▼
RAG检索（只调用一次）
  ├─ Query Rewriting 提取bug模式
  ├─ BM25 + FAISS 多路召回
  ├─ RRF排名融合（替代线性加权）
  └─ Cross-Encoder重排 → top-2相似案例
  │
  ▼
ReAct循环（最多3轮）
  │
  ├─ 1.5B小模型 CoT生成 + 修复代码
  │    输入: buggy code + RAG案例 + (反馈)
  │    CoT四步: Bug定位→根因分析→修复策略→边界验证
  │    输出: <reasoning>...<fixed_code>...</fixed_code>
  │
  ├─ Compile-First验证（ground truth）
  │    compile() 语法检查
  │    exec() + test cases 执行验证
  │
  ├─ 通过? → YES → 返回修复代码，结束
  │
  └─ NO → 32B大模型分析
        输入: 小模型CoT + 修复代码 + 编译报错
        输出: 哪步推理出错（Step 1-4）+ 错误原因 + 修复hint
        反馈给小模型 → 下一轮

3轮都失败 → 记录bad case，进入GRPO训练数据

三、技术选型问答（高频追问）
================================================================================

【Q: 为什么用Qwen2.5不用更新的模型？】
A: 三点原因：
  1. 控制变量：核心贡献是训练方法和架构，不是比拼基座模型
  2. Qwen2.5生态最完整，1.5B/7B/32B三个尺寸满足双模型架构
  3. 架构是model-agnostic的，换Qwen3/DeepSeek-Coder只需改一行路径

【Q: 为什么不直接让大模型修？】
A: 项目目标是训练小模型，大模型是老师不是工人。
  直接让大模型修就失去了训练小模型的意义，且大模型有幻觉。

【Q: 训练框架是什么？】
A: PyTorch + HuggingFace全家桶
  - 模型加载: Transformers (AutoModelForCausalLM)
  - 参数微调: PEFT的LoRA
  - SFT训练: HuggingFace Trainer
  - GRPO训练: TRL库的GRPOTrainer (trl==0.9.4)
  - 单卡A100 40GB，FP16，没用DeepSpeed/FSDP

【Q: ReAct用的什么框架？】
A: 纯自己写的，没用LangChain/LlamaIndex。
  原因：Compile-First需要精细控制循环逻辑，通用框架不好控制。
  参考的是Yao et al., 2022 ICLR原版ReAct的Thought-Action-Observation范式，
  但做了两个改动：
  1. Observation来自编译/测试的真实输出，不是大模型判断
  2. 编译通过就直接跳出，不走完整个循环

【Q: 60%不需要大模型这个结论怎么得来的？】
A: 来自评测集上的ReAct Agent iteration日志统计。
  每个case有完整的迭代记录，统计iteration字段得出。
  60%是第一轮小模型直接通过编译+测试的比例。
  - 第1轮: 约60%
  - 第2轮（带大模型反馈后）: 约20%
  - 第3轮: 约7%
  - 3轮都失败: 约13%

【Q: 复合bug（多bug同时存在）修不了？】
A: 两个原因：
  1. 报错信号有误导性：两个bug共存时，编译器只报第一个触发的错误，
     Bug B被Bug A的症状遮盖，先修A才能暴露B
  2. 1.5B模型同时满足多个约束的能力有限，即使大模型指出了两处，
     小模型要在一次生成里同时修对两处很吃力

【Q: 为什么不加到大模型fallback？】
A: 可以加，实际部署会这么做。
  但评测时分开报：分别展示小模型(82%)、加ReAct(87%)、加LLM fallback(93-95%)，
  这样才能归因到训练方法的贡献，不是大模型本身。

【Q: RAG数据格式是什么？】
A: 每条是JSON对象：
  {
    "bug_type": "off_by_one",
    "original_code": "def binary_search...",
    "fixed_code": "def binary_search...",
    "description": "...",
    "source": "stackoverflow"
  }
  - 596条存在runs/bug_fixes.json
  - original_code + fixed_code + description拼接 → BM25倒排索引
  - 同样的文本 → UniXcoder编码 → FAISS IndexFlatIP
  - bug_type → 倒排索引做type boost

【Q: BM25公式是什么？】
A: BM25(q,d) = Σ IDF(t) × tf(t,d)·(k1+1) / tf(t,d) + k1·(1-b+b·|d|/avgdl)
  k1=1.5, b=0.75
  TF-IDF改进版，k1控制词频饱和度，b控制文档长度归一化

【Q: RRF公式是什么？】
A: RRF(d) = Σ 1/(k + rank_r(d)), k=60
  基于排名融合，不依赖原始分数的量纲
  BM25分数范围0-30，余弦相似度0-1，线性加权需要归一化，RRF直接用排名更鲁棒

【Q: 变量名不同怎么检索？】
A: BM25匹配语法关键词(return, def, for, +, -)，不随变量名变
  FAISS匹配语义embedding，def add(a,b): return a-b 和 def sum(x,y): return x-y
  在向量空间距离很近，因为embedding模型学到的是语义不是字面

【Q: 为什么不用MCTS？】
A: MCTS适合搜索空间大且每步有明确状态转移的场景（如多文件修复、长链路推理）
  我的场景CoT是四步线性推理，搜索空间不大，用MCTS开销不划算

【Q: 为什么用LoRA不用全参数SFT？】
A: 全参数SFT在3000条数据上容易过拟合
  LoRA只训练0.18%参数(~275万)，相当于隐式正则化
  弱基座+小数据+小参数 = 防过拟合的最优组合

【Q: LoRA加载了哪些模块？】
A: target=q_proj, k_proj, v_proj, o_proj（4个模块）
  r=8, alpha=32, dropout=0.1

【Q: GRPO的reward函数怎么算的？六维度】
A: 格式奖励(0.1): 是否有完备的CoT四步结构
  语法奖励(0.1): compile()能否通过
  AST结构(0.15): ast.parse()后节点类型集合的Jaccard相似度 + 函数签名匹配
  测试执行(0.5): pytest通过比例
  语义相似度(0.15): UniXcoder余弦相似度
  过程步骤(0.15): CoT中是否包含Step 1-4的关键词

【Q: 为什么GRPO数据从SFT里抽1200条，不会数据泄露？】
A: 不会。
  SFT学的是"给定buggy code输出fixed code"的能力
  GRPO学的是"给定同一批prompt，区分高reward和低reward的策略"
  虽然见过答案，但GRPO的目标是学会区分，不是学会背答案。
  如果不泄露，那reward=1的样本会淹没reward=0的样本，优势函数全为正，没有梯度。

【Q: 数据量3000条够吗？】
A: 代码修复是narrow task，不需要通用能力
  3000条覆盖8类主诉bug，用LoRA r=8只训练0.18%参数
  加上数据增强（变量重命名、bug类型迁移、风格变换）

【Q: 怎么判断是Python代码？】
A: Python语言验证：长度≥80字符、行数≥4、含def/import/for关键词、排除JS/C++
  compile()语法检查

【Q: MD5去重怎么做的？】
A: 对buggy+fixed拼接字符串做MD5哈希，相同hash认为是重复

【Q: 为什么先compile再考虑是否给大模型看？】
A: compile()是ground truth，零幻觉零成本
  如果直接给大模型看，大模型可能产生幻觉说"代码没问题"
  compile()通过 = 确定代码正确，跳过大模型节省60%算力

【Q: Reflexion架构怎么实现的？】
A: 反思经验存在ReflectionMemory里，格式：
  {"bug_type": "off_by_one", "reflection": "快排pivot选择错误会导致无限递归..."}
  三轮失败后自动生成反思存入记忆
  新任务开始时检索相关历史反思注入prompt

【Q: 什么是UniXcoder？】
A: 微软开源的代码预训练模型，结构：
  - 基于BERT/RoBERTa
  - 动态mask训练（类似BERT MLM）
  - 15种编程语言预训练
  - 支持code search、code completion等任务
  比通用embedding更能理解代码语义（append vs extend在通用模型里被认为相似）

四、训练参数速查表
================================================================================

【数据】
  - 种子数据: 596条
  - 增强后: ~3000条（5倍增强：变量重命名、bug迁移、风格变换）
  - 来源: StackOverflow API + GitHub + 公开种子数据
  - 清洗保留率: 60-70%
  - Bug类型分布:
      wrong_operator 18% (~540条)
      off_by_one     15% (~450条)
      type_error     12% (~360条)
      index_error    10% (~300条)
      condition      10% (~300条)
      wrong_return    8% (~240条)
      null_check      7% (~210条)
      其他           20% (~600条)

【SFT】
  - 模型: Qwen2.5-Coder-1.5B-Instruct（8层，12Q/2KV）
  - 数据: 3000条（buggy→fixed，无CoT）
  - LoRA: r=8, alpha=32, dropout=0.1
  - target: q_proj, k_proj, v_proj, o_proj（4个模块）
  - LoRA参数量: ~275万（占比0.18%）
  - LR: 2e-4
  - Batch: 1，梯度累积2，有效batch=2
  - Epoch: 1
  - 总步数: 3000÷2 = 1500 steps
  - 精度: fp16
  - Loss: 2.5 → 0.8
  - 显存: ~20GB
  - 时间: ~25min（A100）
  - 效果: pass@1 54% → 70%

【GRPO】
  - 策略模型: 1.5B（SFT后）+ LoRA
  - Reward模型: 7B（SFT后，只推理不训练）
  - 数据: 1200条（筛掉全对/全错）
  - LR: 1e-5
  - Batch: 1，梯度累积2，有效batch=2
  - num_generations: 4
  - max_prompt_length: 512
  - max_completion_length: 512
  - KL系数: 0.02
  - 精度: fp16
  - 显存: ~32GB
  - 时间: ~3小时（A100）
  - 效果: pass@1 76% → 82%

【最终效果】
  - SFT: 54% → 70%
  - + RAG: +5%
  - + Compile-First ReAct: +5%
  - 最终: 87%
  - QuixBugs benchmark, Wilson 95% CI [73%, 95%]

【RAG】
  - 知识库: 596条
  - 检索延迟: <50ms
  - top-3准确率: 85%+
  - BM25: k1=1.5, b=0.75
  - RRF: k=60

Q: 介绍一下你的项目
A: 小模型CoT生成+编译验证，过了就结束；没过带报错找大模型分析反馈给小模型重试，
   60% case不需要大模型，pass@1从54%提到87%。

Q: 为什么用GRPO不用PPO/DPO
A: GRPO无需Critic模型，组内相对优势计算，显存省30%。

Q: RAG怎么做的
A: Query改写→BM25+FAISS多路召回→RRF融合→Cross-Encoder重排。

Q: 大小模型怎么协作
A: 小模型CoT生成→编译验证→失败才找大模型分析→反馈指导小模型重试。

Q: 数据量够吗
A: 质量优先+扩展方案（SWE-bench/Git历史挖掘/LLM合成）。

Q: 和SWE-Agent比有什么区别
A: 他们全靠大模型API，我训练小模型+按需升级，算力省60%。

Q: 为什么不用LoRA/QLoRA
A: A100显存充足，LoRA够用；QLoRA有精度损失。

Q: 为什么不用全参数SFT
A: 数据少，全参数容易过拟合；LoRA只训练0.18%参数=隐式正则化。

九、Bad Case分析（必背两类）
================================================================================

SFT后30%的失败：
  - 多步逻辑bug：快排Hoare分区，两处耦合，SFT没见过多样本
  - 数据结构语义：指针引用理解，SFT本质是模式匹配
  - 算法特异性：每种算法正确写法不同，数据覆盖不了
  - 边界条件遗漏：修主逻辑但漏edge case

最终13%的失败：
  - 复合bug（~5%）：两个bug互相遮盖，一轮只暴露一个
  - 语义理解（~4%）：函数意图理解错误，1.5B模型有瓶颈
  - 领域特异性（~3%）：正则/并发/编码问题，训练数据少
  - 上下文不足（~1%）：跨函数/跨文件bug需要repo-level上下文
