## 📋 一、项目概述

### 项目名称
**RL-Enhanced Code Repair Agent with RAG and LLM Critic**

### 核心目标
实现一个完整的代码修复 Agent 系统，整合：
- **GRPO**（Group Relative Policy Optimization）强化学习
- **RAG**（Retrieval-Augmented Generation）检索增强
- **LLM Critic** 大模型裁判奖励系统
- **ReAct** 推理 - 行动决策框架

### 应用场景
- 字节跳动大模型算法岗面试项目
- 代码自动修复（Code Bug Fixing）
- RLHF 对比实验（GRPO vs DPO vs PPO）

## 🏆 六、奖励系统设计

### 混合奖励（Hybrid Reward）

```python
Reward = 格式奖励 (0.2) + 语法奖励 (0.3) + LLM Critic (0.5)
```

#### 1. 格式奖励（0.2）
- 检查 `<reasoning>` 和 `<fixed_code>` 标签
- 快速验证输出格式

#### 2. 语法奖励（0.3）
- Python 语法检查（compile）
- 防止语法错误

#### 3. LLM Critic（0.5）
- 使用 Qwen-plus 评估
- 三个维度：
  - **Correctness** (60%): 是否正确修复
  - **Code Quality** (30%): 代码质量
  - **Minimality** (10%): 简洁性

### LLM Critic Prompt

```
You are a senior software engineer. Please evaluate:

**Bug Description:**
{bug_description}

**Buggy Code:**
{buggy_code}

**Generated Patch:**
{patch}

Evaluate on:
1. Correctness (0-10)
2. Code Quality (0-10)
3. Minimality (0-10)

Respond in JSON format.
```

**解决方案**:
```python
# TRL >= 0.8.0
# DPOConfig 中不使用 beta
# DPOTrainer 中使用 loss_beta

    loss_beta=0.1,  # 在 Trainer 里设置

)
```

💡 四、面试回答（参数选择依据）
Q1: “为什么 SFT 的 learning_rate 比 GRPO 大？”
A:
"SFT 用 2e-5，GRPO 用 1e-5，原因：

SFT（监督学习）：
- 有标准答案（chosen）
- 梯度方向明确
- 学习率可以大一点（2e-5）
- 收敛快（25 分钟）

GRPO（强化学习）：
- 无标准答案，Reward 模型打分
- 梯度方差大（4 个候选的 reward 不同）
- 学习率要小一点（1e-5）
- 收敛慢（3 小时）

我试过 GRPO 用 2e-5：
- reward 波动大（-2.52 → -1.8 → -2.3）
- KL 散度爆炸（>0.5）
- 模型输出乱码

降到 1e-5 后稳定了。"
￼
Q2: “为什么 GRPO 的 group_size=4？”
A:
"group_size 是每个 prompt 生成的候选数：

对比过三个值：
1. group_size=2:
   - 显存：28GB（够用）
   - 多样性：不够（只有 2 个候选）
   - pass@1: 79%

2. group_size=4:
   - 显存：32GB（刚好）
   - 多样性：适中（4 个候选）
   - pass@1: 82%

3. group_size=8:
   - 显存：38GB（接近 OOM）
   - 多样性：好（8 个候选）
   - pass@1: 83%（提升不明显）

选 4 是性价比最高的。"
￼
Q3: “为什么 SFT 的 LoRA r=16，GRPO 的 r=8？”
A:
"LoRA 秩的选择：

SFT（r=16）：
- 学习基本修复格式
- 需要较大容量
- trainable params: 5.0M
- pass@1: 70%

GRPO（r=8）：
- 在 SFT 基础上优化
- 不需要太大容量
- trainable params: 2.2M
- 显存更省（少 2GB）
- pass@1: 82%

我试过 GRPO 用 r=16：
- pass@1: 83%（提升 1%）
- 显存：34GB（多 2GB）
- 训练时间：3.5 小时（慢 30 分钟）

选 r=8 性价比更高。"
￼
Q4: “为什么 gradient_accumulation_steps 不同？”
A:
"SFT: accumulation=4
GRPO: accumulation=2

原因：

SFT：
- 单模型（7B + LoRA）
- 显存占用 28GB
- 还有 12GB 余量
- 可以累积 4 步

GRPO：
- 双模型（1.5B + 7B）
- 显存占用 32GB
- 只有 8GB 余量
- 只能累积 2 步

如果 GRPO 用 accumulation=4：
- 显存：36GB（接近 40GB）
- 可能 OOM
- 所以用 2 步"
￼
Q5: “为什么 warmup_steps 不同？”
A:
"SFT: warmup=10
GRPO: warmup=20

原因：

SFT：
- 监督学习，梯度稳定
- 预热 10 步就够了
- 约占总步数 5%（10/188）

GRPO：
- 强化学习，梯度方差大
- 需要更长预热
- 预热 20 步
- 约占总步数 2.5%（20/800）

我试过 GRPO 用 warmup=10：
- 前 50 步 reward 波动大
- KL 散度突然>0.5

增加到 20 步后稳定了。"

def rule_based_score(code):
    scores = {}
    
    # 1. 语法正确性（20 分）
    try:
        compile(code, '<string>', 'exec')
        scores['syntax'] = 20.0
    except SyntaxError:
        scores['syntax'] = 0.0
    
    # 2. 包含 def（20 分）
    scores['function'] = 20.0 if 'def ' in code else 0.0
    
    # 3. 包含 return（20 分）
    scores['return'] = 20.0 if 'return ' in code else 0.0
    
    # 4. 长度合理（20 分）
    scores['length'] = 20.0 if 10 < len(code) < 500 else 0.0
    
    # 5. 与 buggy 不同（20 分）
    scores['different'] = 20.0 if code != buggy else 0.0
    
    return sum(scores.values())  # 0-100 分

检索机制
1. BM25 稀疏检索：关键词匹配
2. FAISS 稠密检索：语义理解
3. 分数融合：0.4×BM25 + 0.6×FAISS
4. Top-K 排序：返回最相似的 2 个案例
检索示例
查询："Fix the addition function"

BM25 检索：
- 分词：["fix", "the", "addition", "function"]
- 计算 IDF 和 TF
- 返回 BM25 分数

FAISS 检索：
- 生成查询向量（384 维）
- 内积相似度搜索
- 返回 FAISS 分数

分数融合：
combined = 0.4 * bm25_norm + 0.6 * faiss_scores

Top-2 结果：
1. Fix the arithmetic_operator bug (分数：0.952)
2. Fix the multiplication bug (分数：0.760)

问题 1: DPO 训练 loss 爆炸
现象:
loss: 832 → 1482 → 40109
grad_norm: 4616 → 32560 → Infinity
原因:
• 奖励函数计算错误
• tokenizer 不匹配
• 参数配置错误
解决方案:
• 改用 GRPO
• 修复 reward 函数
• 检查数据格式
￼
问题 2: TRL 版本不兼容
现象:
ImportError: cannot import name 'GRPOConfig' from 'trl'
解决方案:
# 安装兼容版本
pip install trl==0.9.4 transformers==4.36.0 peft==0.7.0
￼
问题 3: 显存不足 OOM
现象:
RuntimeError: CUDA out of memory. Tried to allocate 32GB
解决方案:
• 降低 batch_size（2→1）
• 减少 gradient_accumulation_steps
• 用 gradient_checkpointing
• 降低 max_length（512→256）
￼
问题 4: bitsandbytes 错误
现象:
ImportError: cannot import name 'clear_device_cache' from 'accelerate.utils.memory'
解决方案:
# 降级 accelerate
pip install accelerate==0.25.0

2 分钟技术介绍
"我做了 CodeFix-Agent 项目，解决通用大模型在代码场景上下文短、推理慢的问题。

技术方案分 5 个阶段：

1. 数据准备（2 周）：
   收集 596 条高质量 bug 修复数据，包括 LeetCode、QuixBugs、LiveCodeBench。

2. SFT 训练（1 周）：
   7B 模型 + LoRA，训练 188 步（25 分钟），loss 从 5.24 降到 0.87。
   pass@1 从 54% 提升到 70%（+16%）。

3. RAG 索引构建（1 周）：
   BM25+FAISS 混合检索，检索延迟<50ms，准确率 85%+。
   pass@1 额外提升 5%。

4. GRPO 训练（2 周）：
   1.5B 策略 + 7B Reward，训练 800 步（3 小时）。
   reward 从 -2.52 提升到 -1.08（+57%）。
   pass@1 从 76% 提升到 82%（+6%）。

5. ReAct 推理（1 周）：
   1.5B 生成 + 32B 验证，80% 计算在小模型。
   pass@1 达到 89%（+7%）。

总提升：54% → 89%（+35%），验证了 RAG+RLHF 在代码修复场景的有效性。"
￼
5 分钟深入讲解
"技术选型上，我对比了三种 RLHF 方法：

1. PPO：需要 Critic 模型，显存占用高
2. DPO：需要偏好数据，小样本易过拟合
3. GRPO：组内相对优势，无需 Critic，最稳定

最终选择 GRPO，实现细节：
- Group Size=4，每个 prompt 生成 4 个候选
- Advantage Normalization，组内标准化
- KL Penalty=0.02，防止偏离 reference

工程挑战：
1. 显存不足：1.5B+7B 同时加载要 32GB，我用梯度累积解决
2. Reward Hacking：用 KL penalty 防止模型刷分
3. 梯度爆炸：学习率从 5e-5 降到 1e-5，加梯度裁剪

这个项目让我深入理解了 RLHF 的完整流程，
从理论到工程实现都有了实践经验。"
￼
高频问题及回答
Q: 为什么选择 GRPO？
A:
"GRPO 核心优势：
1. 无需 Critic 模型，显存降低 30%
2. 组内相对优势计算，训练更稳定
3. 适合小样本场景（100-500 条）

我的实验中，GRPO 训练最稳定，reward_std 保持 0.35-1.53，
而 DPO 出现了 loss 爆炸（832 → 40109）。"
￼
Q: 数据量够吗？
A:
"对于对比实验是足够的：
- 187 条 SFT 数据覆盖 20+ 类错误
- 400 条 GRPO 数据来自 LiveCodeBench
- 每类平均 9 条，统计显著性足够

关键是质量而非数量。我精心设计了 187 条种子数据，
覆盖 20+ 类错误，然后通过拒绝采样扩充到 596 条。

如果要工业级应用，我会：
1. 爬取 SWE-bench、QuixBugs
2. 使用 LLM 生成合成数据
3. 收集真实 bug 修复案例

但在校招面试场景下，596 条已经足够证明我对 RLHF 方法
的理解和工程能力。"
￼
Q: SFT 和 GRPO 的区别？
A:
"SFT 是监督学习，GRPO 是强化学习：

SFT：
- 有标准答案（chosen）
- CrossEntropy Loss
- 拟合 chosen，学会格式
- pass@1 提升 16%

GRPO：
- 无标准答案，Reward 模型打分
- Policy Gradient Loss
- 优化策略，学会逻辑
- pass@1 提升 6%

SFT 让模型'照猫画虎'（学会格式），
GRPO 让模型'举一反三'（学会逻辑）。
两者结合，pass@1 从 54% 提升到 89%（+35%）。"
