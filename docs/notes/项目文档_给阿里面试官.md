## 一、项目一句话介绍

我做了两件事：

1. **代码修复 Agent**：小模型（1.5B）生成 CoT 推理 + 修复代码，编译器作为 Ground Truth 验证，60% 简单 bug 首轮通过；没过的才发给大模型（32B）分析哪步推理出错，最多 3 轮重试。Pass@1 从 54% 提升到 87%。

2. **会议总结系统**：0.5B 模型 LoRA 微调 + BGE 向量检索 + RAG 历史记忆 + GRPO 六维 Reward 强化学习，幻觉率从 28% 降到 18%。

## 二、系统架构

```
Bug 输入
   │
   ▼
RAG 检索（596条 bug 修复案例，BM25+FAISS 混合检索）
   │
   ▼
小模型（Qwen2.5-Coder-1.5B）生成 CoT 四步推理 + 修复代码
   │
   ▼
编译器 + 测试用例（Ground Truth，零幻觉）
   │
   ├─ 通过（60%）→ 直接输出
   │
   └─ 失败（40%）→ 大模型（32B）分析 CoT 步骤错误
                        │
                        ▼
                  小模型基于 hint 重试（最多 3 轮）
                        │
                        ├─ 成功 → 存入 L3 技能库
                        │
                        └─ 3 轮全失败 → 存入 L2 反思库
```

## 三、核心创新点

### 1. Compile-First 条件升级策略

核心思路：**能用编译器判断的就不用大模型**。

- 简单 bug（60%）：小模型 CoT 后直接编译测试，通过就结束，大模型从未调用
- 困难 bug（40%）：编译失败才升级到大模型，大模型拿着报错 + CoT 分析
- 编译器是 ground truth（零幻觉），节省 60% 大模型算力

### 2. CoT 四步推理结构

```
[Step 1: Bug Identification]        — 定位 bug 类型和位置
[Step 2: Root Cause Analysis]       — 为什么出错（执行路径追踪）
[Step 3: Fix Strategy]              — 最小修改方案
[Step 4: Edge Case Verification]    — 边界条件验证
```

### 3. 三层记忆架构

| 层级 | 名称 | 作用 | 触发条件 |
|------|------|------|----------|
| L1 | 工作记忆 | 当前会话信息（bug 描述、错误信息、测试结果） | 每次请求 |
| L2 | Reflection（守） | 失败教训库，存储 bug_pattern + root_cause | 3 轮全失败 |
| L3 | Hermes Skill（攻） | 成功 SOP 库，抽取可复用修复步骤 | 复杂 case 成功 |

### 4. 六维混合奖励函数（GRPO 训练用）

| 维度 | 权重 | 信号来源 | 衡量 |
|------|------|----------|------|
| 格式 | 0.05 | CoT 结构完整性 | 整体结构 |
| 语法 | 0.10 | compile() 通过 | 整体结构 |
| AST 结构 | 0.15 | 与参考答案结构相似度 | 整体结构 |
| 执行 | 0.30 | 测试用例通过率 | 整体结构（最重要） |
| 步骤级 PRM | 0.25 | 7B 对 CoT 每步打分 | **每一步单独衡量** |
| 语义 | 0.15 | Embedding 余弦相似度 | 整体结构 |

步骤级 PRM：用 7B base model 提取 CoT 各步边界位置的 hidden states，过 PRMHead（Linear→GELU→Dropout→Linear→Sigmoid），每个 step 输出 0-1 分数。FIPO 加权 [0.15, 0.20, 0.30, 0.35]，越靠后的步骤权重越高。

### 5. RAG 混合检索

```
Query → Bug Type 倒排索引（596 条 → ~85 条同类型）
              ↓
        BM25 关键词 + FAISS 向量
              ↓
        RRF 融合（k=60，量纲无关，工业标准）
              ↓
        Cross-Encoder 精排 → Top-3
              ↓
        同类型 Bug Boost ×1.5
```

## 六、项目解决了什么问题

通用大模型在代码场景有三个核心痛点，我逐一解决：

**问题 1：上下文窗口短**
> 通用模型处理长代码时上下文被截断。我的 RAG 通过 BM25+FAISS 混合检索，从 596 条知识库动态召回最相关的 bug 修复案例注入上下文，保证有效上下文密度。

**问题 2：推理质量差（幻觉）**
> 通用模型生成代码时容易幻觉。我的 Compile-First 策略：编译器作为 Ground Truth 验证（零幻觉），通过才输出，不通过才升级大模型。60% 简单 bug 编译器直接判断，不需要模型"自我感觉良好"。

**问题 3：算力成本高**
> 全量大模型推理成本高。我的双模型架构：1.5B 小模型处理 80% 简单 case，只有 40% 复杂 case 才升级大模型。大模型调用减少 60%，延迟降低 44%。

## 七、关键公式

**BM25**：
```
score(q,d) = Σ IDF(w) × (tf × (k+1)) / (tf + k × (1-b + b × |d|/avgdl))
k=1.5, b=0.75
IDF = log((N-n+0.5)/(n+0.5))
```

**RRF**：
```
RRF_score(d) = Σ 1/(k + rank_i(d))  k=60
```

**GRPO KL Penalty**：
```
reward_final = reward_raw - β × KL(π_θ || π_ref)   β=0.1
```

## 八、面试高频问题

### Q：为什么选 GRPO 而不是 PPO/DPO？

> PPO 需要额外 Critic 模型估算 V 值，显存要 40GB，A100 32GB 跑不起来。DPO 需要大量偏好对（几百上千条），187 条数据直接过拟合——loss 从 800 爆炸到 40000。GRPO 同组内互相做 baseline，无需 Critic，显存友好。

### Q：为什么不用困惑度做 reward？

> 困惑度衡量流畅度，不衡量正确性。流畅但错的代码 perplexity 很低，会被错误奖励。步骤级 PRM 提供 4 个 step-level 信号，信息密度是困惑度的 4 倍。

### Q：六维 reward 权重怎么定的？

> 执行奖励权重最高（30%），因为测试用例通过是客观 ground truth。坦率说，这些权重是基于推理设定的初值，没有跑完整的网格搜索。这是局限，未来应该按训练阶段自适应调整。

### Q：沙盒验证是离线还是实时？

> 推理阶段是实时验证：生成 → compile → 测试用例执行 → 返回结果。训练阶段是离线验证：积累样本后离线计算 reward。

### Q：为什么最多三轮重试？

> 统计显示 90% 的可修复 bug 在前 3 轮内完成，第 4 轮及以后的修复成功率接近 0。继续重试浪费算力。超过 3 轮的 case 进入 L2 反思记忆，为下次同类 bug 做准备。

### Q：多个 API Key 共用一个 LLM 时怎么做流量管理？

> 我实现了一个 LLM Router（`api/llm_router.py`），三个核心机制：
>
> **断路器（Circuit Breaker）**：per-model 三态机（CLOSED→OPEN→HALF_OPEN）。连续 3 次失败自动熔断，后续请求跳过该模型，避免雪崩。30 秒后进入 HALF_OPEN 试探，2 次成功则恢复。
>
> **指数退避重试**：失败后等待 base=1s × 2^attempt，上限 60s，加 ±50% jitter 防止惊群效应。自动识别 429/5xx/timeout 等可重试错误。
>
> **模型降级链**：`gpt-4o` 失败 → 自动切 `gpt-4o-mini` → 再失败切 `claude-3-haiku`。零代码改动注册新模型。
>
> 底层还挂了一个 Token Tracker（`api/token_tracker.py`），每个 API Key 独立统计 token 用量和成本，配额达到 80%/95%/100% 分别触发告警。

### Q：断路器为什么要设计成三个状态？两个不行吗？

> 两个不够。CLOSED+OPEN 只能判断"健康/不健康"，但模型可能临时抖动（比如 GPU 占用峰值），直接永久熔断会浪费算力。HALF_OPEN 是一个**试探层**：OPEN 后等 30 秒，然后放几个请求过去试试。如果连续成功，说明模型恢复了；如果又失败，说明是真的挂了，再熔断一轮。这样避免"假痊愈"误判。

### Q：断路器和服务端限流（429）有什么区别？

> 本质不同。429 是**被动防御**——服务端告诉你"别来了"，但你的请求已经发过去了，浪费了这一次调用和延迟。断路器是**主动防御**——客户端侧直接判断模型不健康就跳过去，完全不发请求，减少无效调用。两者互补：我先看断路器状态，断路器通过才发请求，收到 429 才触发退避重试。

### Q：指数退避的 jitter 是什么？为什么要有 jitter？

> 纯指数退避有个问题——所有请求退避时间相同，恢复瞬间会有**惊群效应**：1 万个请求同时在 T=1s 发起，1 万个在 T=2s 发起，瞬时流量暴增 1 万倍，服务端再次 429。我的 jitter 加了 ±50% 随机偏移，T=1s 变成 [0.5s, 1.5s]，把并发请求打散到 1 秒内不同时间点，避免流量尖刺。

### Q：降级链怎么决定模型顺序？按成本还是按能力？

> 两个维度都要考虑。我的链是**能力降序 + 成本升序**：`gpt-4o` 排第一因为能力最强，`gpt-4o-mini` 排第二因为便宜 30 倍但能力差不太多，`claude-3-haiku` 排最后是兜底。实践中先试贵的，贵的失败概率低，平均延迟最低；贵的真挂了才降级到便宜的，总体 cost 和 latency 的期望最优。

### Q：Token Tracker 的配额告警怎么设计的？每请求都判断阈值会不会有性能问题？

> 不会。告警判断只有 O(1) 的比较操作，不涉及任何锁或 IO。真正的性能开销是持久化，但只在**阈值跨越时才写盘**——平时 `record()` 只更新内存字典，定期写一次。告警触发条件是"级别上升"（从 OK→WARNING，或 WARNING→CRITICAL），不是每次都触发，避免告警风暴。

### Q：为什么配额用每日重置而不是累计？

> 因为 LLM API 的配额是服务商按天给的，比如 OpenAI 是每天 $5 的 credit，不重置就用不了。而且面试角度说，每日配额对业务更友好——用户知道自己每天有多少额度，服务商也方便做容量规划。

### Q：重试次数设多少？设太多会不会浪费？

> 设 3 次。超过 3 次还没成功的请求，基本不是网络抖动而是业务逻辑问题（比如 prompt 本身导致模型拒绝回答），继续重试没意义。但关键是我会**区分错误类型**：429（限流）/5xx（服务端错误）/timeout/网络错误是可重试的；400（bad request）/401（认证失败）是不可重试的，直接抛异常不浪费重试次数。

### Q：降级链里后几个模型的能力差很多，回复质量怎么保证？

> 降级不是降能力，是**降成本**。设计上 `gpt-4o-mini` 在大多数代码补全场景和 `gpt-4o` 效果差不多，差距只在复杂推理。只有真正需要 `gpt-4o` 的任务（复杂 multi-step bug），降级后失败才会被记录到 X-pool，探索失败经验。等 `gpt-4o` 恢复后再复用。这就是 DECENTMEM 的 exploit-exploration 平衡——降级是探索，不是放弃。
