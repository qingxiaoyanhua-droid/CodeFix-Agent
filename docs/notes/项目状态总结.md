### 5.6 GRPO损失函数

```python
def grpo_loss(self, log_probs, rewards):
    rewards = torch.tensor(rewards, dtype=torch.float32, device=self.policy_device)
    log_probs = torch.tensor(log_probs, dtype=torch.float32, device=self.policy_device)

    # 组内标准化奖励
    mean_reward = rewards.mean()
    std_reward = rewards.std() + 1e-8
    normalized_rewards = (rewards - mean_reward) / std_reward

    # GRPO核心损失: policy gradient with normalized rewards
    loss = -torch.mean(log_probs * normalized_rewards)
    return loss
```

### 5.7 Reward计算（perplexity方式）

```python
class RewardModel:
    def compute_reward(self, prompt, completion):
        full_text = f"{prompt}\n{completion}"
        inputs = self.tokenizer(full_text, return_tensors="pt").to(self.device)

        with torch.no_grad():
            outputs = self.model(**inputs, labels=inputs["input_ids"])
            loss = outputs.loss

        # perplexity越低，奖励越高
        reward = -loss.item() + 1e-6
        return reward
```

## 六、架构设计要点（面试核心）

### 6.1 系统架构

```
用户输入（有bug的代码）
     │
     ▼
┌─────────────────────────────┐
│  CoT Parser                  │
│  <reasoning>...</reasoning>   │  ← 提取CoT推理步骤
│  <fixed_code>...</fixed_code>│  ← 提取修复代码
└──────────────┬──────────────┘
               │
               ▼
┌─────────────────────────────┐
│  Code Executor              │
│  1. 编译验证（零幻觉）       │  ← Compiler as Ground Truth
│  2. 测试用例执行            │
│  3. 正确则输出，失败则升级   │
└──────────────┬──────────────┘
               │
      ┌────────┴────────┐
      │ 失败？（40%）     │
      ▼                  ▼
┌─────────────┐    ┌──────────────────┐
│ 直接输出    │    │ RAG检索          │
│ (60%简单bug)│    │ BM25+FAISS混合   │
└─────────────┘    │ 召回同类型案例    │
                   └──────────────────┘
                          │
                          ▼
                   ┌──────────────────┐
                   │ 小模型重新生成     │
                   │ 失败→大模型修正   │
                   │ (ReAct Loop)     │
                   └──────────────────┘
```

### 6.2 RAG架构（企业级）

```
Query: "Fix the add function: return a - b"

Step 1: Query Rewriting
  → "wrong_operator arithmetic + - * / operator"

Step 2: Bug Type Index（倒排索引）
  → 缩小搜索空间（596条 → ~85条同类型）

Step 3: Hybrid Search
  → BM25（关键词，60%权重）+ FAISS（语义，40%权重）
  → RRF融合（量纲无关，工业标准）

Step 4: Cross-Encoder Reranking
  → 精排Top-3

Step 5: Type Boost
  → 同类型文档RRF得分×1.5
```

## 七、面试话术（基于真实经历）

### 7.1 "你遇到过最难的问题是什么？"

> "依赖地狱——HuggingFace和ModelScope的版本冲突链。
> HuggingFace Hub 1.5.0 → datasets 4.6.1 → modelscope 1.34.0 → transformers 4.36.0。
> 我试了5个版本组合才找到稳定解。根本原因是：modelscope 1.9.5要求datasets<=2.13.0，但sentence-transformers需要huggingface-hub>=0.20.0。
> **解法**：先用`pip uninstall`清干净，再按顺序安装，用`requirements.txt`锁定版本。"

### 7.2 "GRPO训练怎么做的？"

> "设计了双GPU架构：Policy模型（1.5B）放GPU0生成候选，Reward模型（7B）放GPU1计算perplexity奖励。
> 但实际遇到了两个问题：
> 1. **OOM**：GPU1被其他用户占用了26GB，无法kill（权限不足）。改成了单GPU+4bit量化+梯度累积。
> 2. **Tokenizer不存在**：降级transformers后Qwen2Tokenizer消失。查了6种方案，最终用`local_files_only=True`绕过。
> GRPO核心是组内标准化奖励 + KL散度约束 + 6维奖励塑形。"

### 7.3 "BadCase怎么分析的？"

> "10条测试中，sum_list和divide始终通不过（初始化类bug）。
> **根因**：模型对`total=0`和`total=1`这种细粒度差异容易混淆。
> **改进**：
> 1. 数据增强：构造更多初始化类样本
> 2. 提示词优化：强制要求模型先分析变量初始值
> 3. RAG优先召回同类型bug修复案例"

### 7.4 "ReAct架构为什么能降低延迟？"

> "关键设计是编译优先+条件升级：
> - 60%的简单bug（小模型直接修对）→ 直接输出，无需大模型
> - 40%的复杂bug → 才调用大模型做错误归因
> 实测：大模型调用减少60%，平均延迟降低44%。
> 原理：Compiler作为Ground Truth，零幻觉。"
