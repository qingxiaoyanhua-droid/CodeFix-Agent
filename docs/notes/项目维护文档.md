## 三、CoT 四步（对应六维Reward的衡量对象）

CoT 四步不是随意分的，每一步对应六维 Reward 的一个子维度：

```
[Step 1: Bug Identification]     → 格式奖励中的步骤完整性
[Step 2: Root Cause Analysis]   → 步骤级PRM打分
[Step 3: Fix Strategy]           → 步骤级PRM打分 + AST相似度
[Step 4: Edge Case Verification] → 步骤级PRM打分 + 语义相似度
```

**四步为什么够用**：
- 覆盖了代码修复的核心推理链：定位→归因→策略→验证
- 小模型能力有限，更多步数会导致推理链断裂
- 实验验证：5步/6步没有显著提升

## 四、六维Reward体系（CoT全链路覆盖）

六维 Reward 衡量的是 CoT 推理链的完整性和正确性，而非代码的独立属性：

| 维度 | 权重 | 衡量什么 | 具体计算 |
|------|------|---------|---------|
| **格式奖励** | 0.05 | CoT结构完整性 | `'<think>'`、`'</reasoning>'`各0.25；每个`[Step i:]`各0.125；`<fixed_code>`各0.25；4步全齐额外+0.2 |
| **语法奖励** | 0.10 | 代码能否编译 | Python `compile()` 通过=1.0，SyntaxError=0.0 |
| **AST结构奖励** | 0.15 | 修复代码结构相似度 | Jaccard(节点类型集合)×0.6 + 函数签名匹配×0.4 |
| **执行奖励** | 0.30 | 测试用例通过率（最重要） | 通过数/总数，ACES风格按bug类型加权（逻辑bug×1.2，语法bug×0.6） |
| **步骤级PRM奖励** | 0.25 | CoT每步推理质量 | 7B模型对Step1-4逐步打分，FIPO加权（Step4权重最高0.35，Step1最低0.15） |
| **语义相似度奖励** | 0.15 | 修复代码与参考答案语义相似度 | sentence-transformers embedding cosine similarity |

### 格式奖励详细逻辑

```python
# cot_format_reward 的打分规则
score = 0.0
if '<think>' in text: score += 0.25      # 有思考开始标签
if '</think>' in text: score += 0.25      # 有思考结束标签
for i in range(1, 5):
    if f'[Step {i}:' in text: score += 0.125  # 每个步骤标签
if '<fixed_code>' in text and '</fixed_code>' in text:
    score += 0.25                          # 有代码块
# 完整4步CoT额外奖励
if has_all_4_steps and '`' in text:
    score += 0.2                           # 结构完整额外+0.2
```

### 执行奖励的ACES加权

不同 bug 类型的测试有不同的区分能力（discriminative power）：

```python
TEST_DISCRIMINATIVE_WEIGHTS = {
    # 高区分能力（正确vs错误差异明显）
    "logic_error": 1.2,
    "off_by_one": 1.2,
    "initialization": 1.1,
    "boundary": 1.1,
    "null_pointer": 1.1,
    "wrong_operator": 1.0,
    "missing_check": 1.0,
    "type_error": 0.9,
    # 低区分能力（多数正确代码都pass）
    "syntax_fix": 0.6,
    "style": 0.5,
    "optimization": 0.5,
}
```

### FIPO步骤位置加权

FIPO（Future KL）原理：越靠后的推理步骤对最终答案影响越大，所以权重递增：

```python
STEP_WEIGHTS = [0.15, 0.20, 0.30, 0.35]  # Step4权重最高
# Step1-2是推理过程，不直接决定答案
# Step3（Fix Strategy）决定最终代码长什么样
# Step4（Edge Case）是最后一道防线
```

### PRM（Process Reward Model）核心原理

借鉴 OpenAI "Let's Verify Step by Step"：

- **困惑度的问题**：衡量流畅度，不衡量正确性。流畅但错的代码 perplexity 很低，会被错误奖励。
- **PRM 的优势**：1个 sample 产生 4 个 step-level 信号，信息密度是困惑度的 4 倍
- **实现**：7B 模型对 CoT 四步逐步打分（generation 模式，不是 forward pass）

```python
class PRMHead(nn.Module):
    """对每个step的hidden state做二分类：正确/错误"""
    def __init__(self, hidden_dim=4096, num_steps=4):
        self.classifier = nn.Sequential(
            nn.Linear(hidden_dim, hidden_dim // 2),
            nn.GELU(),
            nn.Dropout(0.1),
            nn.Linear(hidden_dim // 2, 1),
            nn.Sigmoid()
        )
```

## 六、GRPO强化学习训练

### 为什么选 GRPO

| 方法 | 问题 |
|------|------|
| PPO | 需要额外 Critic 模型估算 V 值，显存要 40GB，A100 32GB 跑不起来 |
| DPO | 需要大量偏好对（几百上千条），187 条数据直接过拟合，loss 从 800 爆炸到 40000 |
| **GRPO** | 同组内互相做 baseline，无需 Critic，显存友好 |

### GRPO 核心公式

```python
# 组内标准化奖励
mean_reward = rewards.mean()
std_reward = rewards.std() + 1e-8
normalized_rewards = (rewards - mean_reward) / std_reward

# GRPO 核心损失
loss = -torch.mean(log_probs * normalized_rewards)

# 加上 KL 散度约束防止策略崩溃
```

### 双模型架构

```
Policy模型: Qwen2.5-Coder-1.5B（生成候选修复）
Reward模型: Qwen2.5-Coder-7B（SFT后，做步骤级PRM打分）
```

### 训练参数

```python
BATCH_SIZE = 1
NUM_GENERATIONS = 4
LEARNING_RATE = 5e-6
NUM_EPOCHS = 1
MAX_LENGTH = 512
GRADIENT_ACCUMULATION_STEPS = 4
LoRA: r=8, alpha=16, modules=[q_proj,k_proj,v_proj,o_proj]
量化: 4bit NF4, 单GPU可跑
```

## 九、面试高频问题回答

### Q: "为什么CoT用四步而不是更多？"
> "四步足够覆盖代码修复的核心推理链：定位bug→分析根因→制定策略→验证边界。再多会导致小模型推理链断裂，反而降低准确率。这是我实验对比得出的，5步和6步效果没有显著提升。"

### Q: "GRPO为什么用6个维度而不是困惑度？"
> "困惑度衡量流畅度，不衡量正确性。流畅但错的代码PP很低，会被错误奖励。我设计了六维度，其中最关键的是步骤级reward——借鉴OpenAI 'Verify Step by Step'，让7B对CoT每步独立打分，1个sample产生4个信号，信息密度是困惑度的4倍。"

### Q: "怎么解决小模型生成质量问题？"
> "两个机制：(1) 编译验证——编译器结果作为Ground Truth，零幻觉；(2) 条件升级——没通过的case带CoT和报错找大模型分析，大模型指出哪步推理出错，反馈给小模型重做。"

### Q: "RAG为什么不用纯语义检索？"
> "代码bug有明显的关键词模式——'wrong_operator'、'off_by_one'等。纯语义检索对这类pattern不敏感。我用BM25捕捉关键词(60%权重)+FAISS捕捉语义(40%权重)+RRF融合(量纲无关)+Cross-Encoder精排，85% top-3准确率。"

### Q: "六维Reward怎么保证不 reward hacking？"
> "三个机制：(1) 六维互相制衡——单维hack其他维会拖后腿；(2) KL散度约束——限制策略偏离太远；(3) 动态采样——DAPO的思想，生成结果差异大（方差大）的prompt多练。"

### Q: "FIPO步骤权重怎么定的？"
> "这个确实是我根据代码修复任务的特性主观设定的，但有两层依据：第一层是FIPO论文的理论——后期步骤对最终答案影响更大；第二层是代码修复的直觉：Step3（Fix Strategy）决定最终代码，Step4（Edge Case）是最后防线，权重递增是原则性方向。"

### Q: "RM的排序是怎么保证的？"
> "有两个RM：Reward模型（7B）负责PRM步骤级打分，这个是二分类有ground truth；还有一个是偏好学习的RM。两个都存在主观性，解决方案是六维reward互相验证——执行奖励（测试用例通过）是客观ground truth，其他五维是辅助信号。"

### 弱点一：选型理由说不清

| 你说过的 | 面试官追问 | 问题 |
|---------|-----------|------|
| ChromaDB "方便测试" | FAISS 也是一行代码，为什么要返工？ | 不知道 ChromaDB vs FAISS 的核心区别 |
| BGE-M3 "效果最好" | BGE 系列的最新版本核心突破是什么？ | 直接说不了解 |
| 0.35 置信度阈值 | 怎么确定的？BERT 预测错了怎么办？ | 拍脑袋定的，没有依据 |
| 六维权重 "做了消融实验" | 训练过程中权重会变吗？ | 全程不变——反而是弱点 |

**需要补**：BGE-M3 vs BGE-Large vs SGPT 的区别；FAISS 四种索引（HNSW/IVF/Flat/IMI）的适用场景；置信度阈值确定的标准方法。

### 弱点二：公式/原理说不全

| 工具 | 应该知道的 |
|------|-----------|
| BM25 | 完整公式：score = Σ IDF × (tf × (k+1)) / (tf + k × (1-b + b × dl/avgdl))，k=1.5~2.0，b≈0.75 |
| HNSW | 时间复杂度 O(log N)；EF/ML 参数控制精度-速度权衡；召回率非 100%（局部最优） |
| RoPE | 旋转位置编码的公式；为什么能处理长序列；和绝对/相对位置编码的区别 |
| GQA | KV cache 节省比例 = num_kv_heads / num_q_heads；MQA 是 GQA 的特例 |
| KL散度 | 在GRPO里怎么融进loss的——不是单独的项，而是加在 reward 上减去 β×KL |

### 弱点三：数字细节前后不一致

**六维 reward 在两个项目里权重不同：**

会议项目（百度/WXG二面自述）：
- 准确性 30% + 忠实度 20% + 流畅度 15% + 相关性 15% + 结构化 15% + 长度 5%

代码项目（简历/项目文档）：
- 格式 0.05 + 语法 0.10 + AST 0.15 + 执行 0.30 + 步骤级 0.25 + 语义 0.15

**这是两个不同的六维，不是一套！** 会议项目是总结任务（忠实度/流畅度），代码项目是修复任务（AST/执行/步骤级）。面试时要明确区分，不能混用同一套权重描述。

### 弱点四：沙盒验证说成离线

面试官（WXG一面）问："代码执行是在训练过程中实时跑，还是离线做的？"
你答："离线跑的验证。"

**更好的回答**：训练过程中用的是**离线验证**积累样本，但推理阶段是**实时验证**——小模型生成 → compile 编译 → 测试用例执行 → 通过则输出/不通过则反馈大模型。编译和测试用例是实时运行的，不是离线。

### 弱点五："未来要做"说得太多

每次说不清就转向"未来我会做"——面试官明显感觉到这是在掩盖知识空白。

**改法**：把"未来要做"改成"目前方案及其trade-off"。
- 别说"BERT 还没训练好"——说"当前用置信度阈值兜底，阈值 0.35 是经验值"
- 别说"未来做VL模型"——说"当前只处理文本，PPT/白板内容需要人工标注，这是当前方案的核心局限"

### 弱点六：CoT 四步没有量化依据

面试官可能问："为什么四步不是五步六步？"

**准备好的量化回答**：
> "我在实验里对比了 3/4/5/6 步的 pass@1：3 步时很多 bug 定位和根因混在一起，Step 2 经常溢出；5 步和 6 步没有显著提升，但小模型的上下文窗口和推理能力不足以支撑更多步，反而导致 CoT 断裂。所以 4 步是精度和召回的平衡点。"

### 弱点七：不了解主流工具的边界

| 工具 | 面试官问过什么 | 你应该知道的 |
|------|-------------|------------|
| ChromaDB | vs FAISS 的本质区别 | ChromaDB 是嵌入式向量库，FAISS 是高效的向量检索库；ChromaDB 底层也是 FAISS；ChromaDB 有 metadata filtering，FAISS 需要自己实现 |
| BM25 | 完整公式 | 见上 |
| HNSW | 搜索复杂度、EF参数、召回率 | O(log N)；EF 越大越准越慢；非100%召回 |
| HaluE | 是什么 | Hybrid LLM Evaluation——多维度评估幻觉，结合 NLI、问答等多种方法 |
| OpenChat | 四层记忆 | 短期记忆（2天）、长期记忆（disk）、环境感知、工作内存 |
| BGE | 最新版本核心突破 | BGE-M3：多语言、105种语言、向量维度1024；dense+ Colbert hybrid |

## 十七、快速背诵清单（面试前过一遍）

### 公式背诵

```
BM25: score = Σ IDF(w) × (tf × (k+1)) / (tf + k × (1 - b + b × dl/avgdl))
  其中 IDF = log((N - n + 0.5) / (n + 0.5))
  k=1.5~2.0，b≈0.75

HNSW: 搜索复杂度 O(log N)
  EF_construction 越大，构建越慢，召回越高
  EF_search 控制精度-速度权衡

KL散度（GRPO中）:
  reward_final = reward_raw - β × KL(π_θ || π_ref)
  β 一般设 0.1，训练初期可以warmup到0.1

FIPO步权重: [0.15, 0.20, 0.30, 0.35]（Step越靠后权重越高）
```

### 数字细节背诵

```
ChromaDB: metadata filtering + 时间衰减（半衰期7天，sqrt拉平）
BGE-M3: 1024维，105种语言，dense+ Colbert hybrid
GRPO: group=4, clip=1.2, β=0.1
LoRA: r=8, α=16, target_modules=[q_proj,k_proj,v_proj,o_proj]
CoT: 4步；实验对比了3/4/5/6步选4步最优
pass@1: 54%→72%→87%（SFT→+RAG→+GRPO）
```

### 面试话术转换

| 不要说 | 改成 |
|--------|------|
| "未来要做" | "当前方案有这个局限，trade-off是XXX" |
| "方便/简单" | "因为在这个规模下，延迟<200ms就够了，FAISS的额外复杂度不必要" |
| "我试了很多" | "根据数据集规模和精度要求，选择了XXX，实验对比YYY" |
| "不确定/没算过" | "这个我没有精确数据，但根据经验大概是XXX" |
| "直接调用API" | "集成了BGE-M3预训练模型，dimension=1024" |
