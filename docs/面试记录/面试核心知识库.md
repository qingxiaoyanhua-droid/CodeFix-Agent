## 一、项目架构核心设计

### 1.1 Compile-First 验证策略

**核心理念**：编译器 + 测试用例 = ground truth，零幻觉。

```
小模型生成代码
     ↓
  编译通过？
  ├── 通过 → Ruff 检查代码风格
  │         ├── 有问题 → 轻量优化 pass（无需大模型）
  │         │           ↓
  │         │      通过 → 输出 ✓
  │         └── 无问题 → 输出 ✓
  └── 不通过 → 编译报错发给大模型
                ↓
            大模型反馈给小模型重试（最多3轮）
```

### 1.2 大小模型协作架构（演进版）

面试中讨论出的更优设计：

```
大模型：分析 bug 在哪里 → 告诉小模型"第3行有问题"
         ↓
小模型：只修复第3行（精准修改）
```

- **大模型只给定位指令**，不生成完整代码
- **小模型只执行修改**，不需要理解整体函数
- **代码隐私自然解决**：完整代码始终只在小模型本地
- 大模型看到的是：`第3行: while lo <= hi` → `off-by-one 错误`

### 1.3 六层安全沙箱

详见 `shell_sandbox.py`，6 层防护：
1. 语言白名单
2. 危险模式检测
3. subprocess 隔离
4. 临时目录隔离
5. 资源限制（CPU/内存/时间）
6. 退出码验证

### 1.4 代码隐私问题（面试高频追问）

**分层处理**：

| 场景 | 代码泄露风险 | 解决方案 |
|------|------------|---------|
| 有完整测试用例 | 低 | 小模型+编译器解决大部分 |
| 有复现步骤 | 中 | 堆栈信息足够，不发完整代码 |
| 只有描述无用例 | 高 | 大模型分析 → 脱敏后发送 |

**可以安全脱敏的**（不影响分析）：
- 变量名 → `foo` / `bar`（bug 分析依赖结构，不依赖命名）
- 函数名 → `func_1` / `func_2`
- 注释 → 全删

**不能脱敏的**：
- 控制流和循环边界（off-by-one 依赖边界条件）
- 数据类型和容器结构（越界要看列表长度）
- 错误堆栈（定位根因的核心）
- 测试用例输入输出（ground truth）

## 二、5层渐进压缩（CompactionPipeline）

详见 `state_manager.py`。

| 层 | 名称 | 触发条件 | 具体手段 |
|----|------|---------|---------|
| L1 | Budget Reduction | 单个工具输出 > 8000 字符 | 截断：保留开头+末尾各 2000 字符 |
| L2 | Snip | 累计历史 > 30000 字符 | 裁剪低价值片段 |
| L3 | Microcompact | token 估算 > 40000 | 合并重复结果 |
| L4 | Context Collapse | token 估算 > 60000 | 时间投影摘要 |
| L5 | Auto-compact | token 估算 > 80000 | LLM 语义摘要 |

**设计原则**：按需压缩，成本递增。前面层能搞定就不触发后面层，每层都有日志可回溯。

## 三、L3 技能淘汰机制

### 3.1 双阈值淘汰策略

```
技能质量分 = hit_count × success_rate
```

同时卡两个维度：
- **命中阈值**：技能必须被调用过足够多次才有统计意义（建议 N >= 5）
- **成功率阈值**：调用多但成功率太低应淘汰（建议 X >= 60%）

### 3.2 阈值怎么定

**看数据分布，不拍脑袋**：
1. 跑历史数据，看现有技能的 hit_count 和 success_rate 分布
2. 找自然分界点（双峰分布）
3. 在分界点附近取整
4. 上线后持续观察"误杀率"

### 3.3 贝叶斯更新（防误杀）

不用简单的"N次失败就删除"，维护 Beta 分布：
```
skill_score ~ Beta(alpha + successes, beta + failures)
当 P(success_rate > 0.5 | data) < 阈值时淘汰
```
失败 1 次不会立刻淘汰，连续失败才快速压低概率。

### 3.4 分阶段阈值

```
上线初期（< 100 个技能）：命中 >= 2，成功率 >= 30%
成长期（100~500）：命中 >= 5，成功率 >= 50%
成熟期（> 500）：命中 >= 10，成功率 >= 60%
```

### 3.5 技能相似了要不要合并？

**三层判断**：
1. **结构相似度**（embedding cosine > 0.85 → 进入候选）
2. **Fix strategy 是否冲突**（"预防性检查" vs "异常捕获"不能合并）
3. **触发词重叠率**（overlap > 0.7 才合并）

**合并策略**：抽象成父子技能分层管理，不直接丢弃。

### 3.6 检索时分组

两阶段检索 + 分组去重：
```
Query → bug_type 倒排索引过滤 → 组内质量排序 → 跨组多样性采样（每组最多选 2 个）
```

### 3.7 技能执行监控

三层监控：
1. **执行层**：每次调用的输入输出和耗时
2. **技能层**：每个 skill 的成功率和延迟趋势
3. **系统层**：整体覆盖率变化

告警：成功率持续下降比绝对值低更重要。

## 四、强化学习核心概念

### 4.1 Wilson 置信区间

**为什么用**：代码评测样本量小（几十道题），正态近似会算出负数或超过 100%，Wilson 更诚实。

**核心思想**：把实测比例往 50% 拉一拉再算范围，测得越少拉得越多。

**面试一句话**：
> "Wilson 通过在中心项里加一个 z²/2n 的惩罚项，把实测比例往 50% 收缩，本质上是在贝叶斯视角下用 Jeffreys 先验。公式是 C = (p̂ + z²/2n) / (1+z²/n)，区间 = [C - S, C + S]。"

### 4.2 VPO 强化学习

**核心**：不同维度的奖励系数是随机采样的，没有更复杂的机制。

**Dir(1) 均匀分布**：
```
这次抽到 w = (0.8, 0.2, 0.0) → 主要关注测试通过率
下次抽到 w = (0.1, 0.9, 0.0) → 主要关注编译成功率
```
同一个候选集，不同权重选中不同的 y → 梯度信号多样性自然涌现。

**"随机"是刻意选择**：
1. 随机保证了无偏
2. 随机保证了多样性探索
3. 结合多答案后，diversity 是 max 操作的自然结果

## 五、真实仓库评测方案

### 方案一：GitHub fix commit（最准确）

每个 fix commit 都是天然的 (buggy_code → fixed_code) 对：
```bash
git log --all --oneline --grep="fix" -- "*.py" > fix_commits.txt
```
推荐仓库：Django / Flask / requests

### 方案二：自己注入 bug

给正确实现的函数注入常见 bug：
- off-by-one 循环边界
- 越界访问
- 错误操作符
- 空指针未检查

### 方案三：回归测试套件

真实仓库有完整测试套件，修复前有 N 个 FAIL，修复后全 PASS 即成功。

### 方案四：LLM 把 bug 描述映射到测试子集

只跑相关测试，不跑全量，节省时间。

## 六、工程化问题汇总

### 6.1 面试暴露的缺失点（按优先级）

| 优先级 | 缺失点 |
|--------|--------|
| P0 | AST 定位模块文档写了但没集成 |
| P0 | 项目边界主动说清楚，不能等追问 |
| P0 | 仓库级代码量大时定位策略一片空白 |
| P0 | 代码实际运行验证是离线的 |
| P1 | GRPO reward 权重怎么定（消融实验） |
| P1 | Pass@k 和 Pass@1 区别说不清 |
| P1 | 为什么选这个方案讲不清楚 |
| P1 | 产品思维缺失 |
| P2 | BM25 公式细节 |
| P2 | LoRA 为什么能防止过拟合 |

### 6.2 项目上线问题

**不要说**：还没上线、比较遗憾。

**要这样说**：
> "技术上是 Ready 的——FastAPI 接口通了、Docker 镜像可以打、整条调用链路跑通了。只是没有真实用户和业务场景。"

**收尾**：
> "这也让我意识到，做 AI 应用不只是调 API 跑通 demo，更重要的是想清楚谁来用、怎么用。这也是我来面这个岗位的原因。"

### 6.3 代码隐私问题

**三个解法**：
1. **编译优先**：60% 简单 bug 不需要大模型
2. **本地部署**：Ollama / vLLM 在内网跑
3. **脱敏**：只提取行号+错误类型，不发完整代码

### 6.4 前端实现

纯 HTML + CSS + JavaScript（vibe coding），不需要框架：
- 状态管理：纯 JS 对象 + localStorage
- API 调用：fetch
- 消息渲染：拼 HTML 字符串 + insertAdjacentHTML
- 三种模式：简单描述 / 完整代码 / 上下文续接

用户画像和 L2/L3/RAG **不冲突**：
- L2/L3 是推理依据
- RAG 是经验库
- 用户画像只影响 RAG 结果排序

## 七、面试话术精华

### 7.1 STAR 法则框架

| 部分 | 内容 |
|------|------|
| Situation | L3 技能库积累了几十个修复 SOP，成功率只有 20% |
| Task | 设计机制自动淘汰低质量技能 |
| Action | 双阈值淘汰（hit >= 5 且 success >= 60%）+ 贝叶斯更新 |
| Result | 平均成功率从 58% 提升到 81% |

### 7.2 Docker vs K8s

**Docker 必装**：完整走一遍 build → run → rm → images → ps，20 分钟够了。

**K8s 不需要学**：门槛高，有运维团队，不需要你来管集群。

### 7.3 Python 常见追问

- 装饰器（`*args` `**kwargs`）
- 上下文管理器（`with` 原理）
- 生成器（yield vs return，内存优势）
- 深拷贝浅拷贝（画图）
- async/await（GIL 的关系）

### 7.4 面试官问"你觉得有没有团队协作经验"

> "学校课程项目里有过团队合作，这个 CodeFix-Agent 是我课余自己做的，主要是想深入理解 Agent 系统的设计和实现。"

**个人项目是优势**，不是劣势。

### 7.5 面试官问"项目上线没"

> "技术上是 Ready 的。FastAPI 接口通了、Docker 镜像可以打、整条调用链路跑通了。缺少的是真实用户反馈。这也是我来这个岗位的原因——真正在有业务场景的环境里把 AI 应用落地。"

### 7.6 面试官说"运维了解的并不多"

> "确实这是我最明显的短板，但我知道这是最好补的。Docker 删容器/镜像、日志查看、进入容器这三件事我能回答。"

## 八、面试心态

- **技术深度**：不是靠背，是靠落地够多、想得够深
- **诚实**：主动承认不足 + 说明下一步，比硬撑要好
- **态度**：不是"我需要你培养"，而是"我能给你贡献"
- **项目边界**：主动说清楚，不能等追问发现
- **工程化思维**：测试/门禁怎么自动化，比"代码能不能写"更重要
