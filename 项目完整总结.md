## 一、项目概述

### 1.1 项目定位

**目标**：构建一个通用的 AI 代码修复系统，解决通用大模型在代码修复场景中上下文短、推理慢、成本高的问题。

**核心思路**：不依赖大模型做"幻觉式"修复，而是用编译器 + 测试用例作为 ground truth，先让小模型尝试，小模型搞不定再动用大模型。同时配合 RAG 检索、记忆系统、GRPO 训练形成闭环。

### 1.2 核心技术指标

| pass@1 | 54% | 87% | +33pp |

| 平均延迟 | - | -44% | 降低 |
| 大模型调用次数 | - | -60% | 减少 |

```
└─────┬────┘          └──────┬───────┘
      │                      │

                 ▼
┌─────────────────────────────────────────────┐
│         Compile-First 验证                   │
│  • Python 编译器检查语法                      │

└────────────────┬────────────────────────────┘
                 │
        ┌────────┴────────┐
        ▼                 ▼
   ┌─────────┐      ┌─────────────┐
   │ 通过 ✓  │      │ 失败 → 反馈  │
   └─────────┘      └──────┬──────┘
                           │
                     最大 3 轮重试
```

### 2.2 验证流程（Compile-First）

```
小模型生成代码
     ↓
  编译通过？
  ├── 通过 → Ruff 检查代码风格
  │         ├── 有问题 → 轻量优化 pass（无需大模型）
  │         │           ↓
  │         │      通过 → 输出 ✓
  │         │
  │         └── 无问题 → 输出 ✓
  │
  └── 不通过 → 编译报错发给大模型
                ↓
            大模型反馈给小模型重试（最多3轮）
```

**核心设计理念**：编译器 + 测试用例 = ground truth，零幻觉。大模型只在需要复杂推理时才介入。

### 2.3 Ruff 质量检查

- **位置**：编译通过后自动触发
- **作用**：检查代码风格与可读性问题（未使用变量、未导入、命名不规范等）
- **优势**：毫秒级，本地规则匹配，无 LLM 幻觉风险
- **结果**：有问题触发轻量优化 pass，不需要调用大模型

## 三、核心模块详解

### 3.1 ReAct + CoT 调度（`cot_react_agent.py`）

主模型负责：
1. **AST 解析**：分析代码结构，定位函数边界
2. **动态路由**：根据 bug 复杂度决定用小模型还是大模型
3. **CoT 推理**：强制小模型按 4 步思考（识别 Bug → 根因分析 → 修复策略 → 边界检查）
4. **RAG 检索**：结合 DECENTMEM 双池记忆检索相似案例

### 3.2 AST 代码处理器（`ast_code_processor.py`）

- 基于 Python 内置 `ast` 模块
- **字节偏移替换**：不依赖行号，用字节偏移精确定位函数
- 解决行号偏移导致的修复错误问题

### 3.3 三层记忆系统（`dual_pool_memory.py`）

- **L1 上下文压缩**：当前任务上下文，token 级别压缩
- **L2 反思记忆**：失败经验总结，时间衰减权重，每周清理低权重碎片
- **L3 技能记忆**：成功修复 SOP，按 bug 类型组织
- **E-pool / X-pool（DECENTMEM）**：E-pool 存成功轨迹，X-pool 存候选策略，路由器按相似度动态选择

### 3.4 RAG 检索（`enterprise_rag_pipeline.py`）

- **混合检索**：BM25 + FAISS 向量检索
- **融合策略**：RRF（Reciprocal Rank Fusion）
- **重排**：Cross Encoder 二阶段重排，准确率 85%+
- **延迟**：< 200ms

### 3.5 GRPO 训练（`experiments/grpo/`）

训练脚本系列：
- `grpo_codefix_train_cot_prm.py`：GRPO + CoT + Process Reward Model
- `grpo_codefix_train_decentmem.py`：GRPO + DECENTMEM
- `grpo_codefix_train_with_critic.py`：GRPO + Critic 模型
- `grpo_codefix_train_v2.py` / `grpo_train.py`：其他 GRPO 变体

**六维 Reward**：
1. 格式奖励：输出格式正确性
2. 语法奖励：代码可编译性
3. AST 奖励：语法树结构正确性
4. 执行奖励：测试用例通过率
5. 过程奖励：修复步骤合理性（PRM）
6. 语义奖励：代码语义一致性

### 3.6 安全沙箱（`shell_sandbox.py`）

6 层安全防护：
1. 语言白名单
2. AST 解析危险检测（Python）+ 正则黑名单（Shell）
3. subprocess 隔离
4. 临时目录隔离 + 自动清理
5. 资源限制（超时、输出大小）
6. 退出码验证 + 编译输出路径守卫（防止写到沙箱外）

**改进说明**：Python 危险调用检测由正则升级为 `ast.parse` 语法树分析，精确识别 `os.system`、`subprocess.Popen`、`ctypes`、`eval` 等危险结构；编译型语言二进制输出路径强制限制在 `work_dir` 内，防止 `gcc -o /etc/x` 等路径穿越攻击。

### 3.7 API 服务（`api_server.py`）

- FastAPI 构建
- API Key 鉴权
- 滑动窗口限流（30 req/min/key）
- JSONL 审计日志

## 八、设计亮点

1. **Compile-First**：编译器作为 ground truth，零幻觉
2. **大小模型协作**：小模型处理简单 case，大模型只处理复杂 case
3. **Ruff 兜底**：风格问题本地规则解决，不花 Token
4. **DECENTMEM 双池**：按成功/失败分离记忆池，防止脏数据
5. **六维 Reward**：多粒度奖励防止 Reward Hacking
6. **6 层沙箱**：安全执行不可信代码
